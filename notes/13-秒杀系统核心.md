## 一、超卖防护（面试必问 TOP 1）

### Q1: 什么是超卖？为什么会超卖？怎么防？

**考察**：能不能从根因讲起，清晰地说出完整链路

> 面试官问超卖，**先讲根因再讲方案**，不要上来就甩"三层防护"！

**第一步：先讲清楚超卖的根因 -- Check-then-Act 竞争**

超卖的本质是**并发时序下"检查"与"执行"的非原子性**。想象这个物理场景：

```
库存只剩最后 1 个

时刻 1:  线程 A 查库存 -> 看到 stock = 1 -> 认为"有货"
时刻 2:  线程 B 查库存 -> 也看到 stock = 1 -> 也认为"有货"
时刻 3:  线程 A 执行扣减 -> stock 变成 0
时刻 4:  线程 B 也执行扣减 -> stock 变成 -1  !! 超卖了！
```

问题出在哪？**"查"和"扣"之间有时间窗口**，这个窗口里别的线程也查到了旧值。所以解决超卖的核心就一句话：**把"查库存"和"扣库存"变成一个绝对的原子操作**。

**第二步：三层防护方案**

三层防护，层层兜底，像俄罗斯套娃一样套着：

```
第一层：Redis Lua 脚本（挡掉 99% 的无效请求）
  ↓ 只有真正抢到的请求才往下走
第二层：RabbitMQ 异步削峰（保护 MySQL 不被高并发打爆）
  ↓ Consumer 慢慢消费
第三层：MySQL 行锁（最后的保险丝）
  WHERE seckill_num > 0  ← 即使前两层出问题，DB 也不会超卖
```

为什么 Lua 能解决 Check-then-Act？因为 Redis 是单线程执行 Lua 脚本的——整段"查库存 + 扣库存 + 标记用户"在一次执行中完成，**物理上不可能被其他线程插入**。这就从根本上消灭了并发时序冲突。

**打个比方**：第一层是保安在门口查票，大部分没票的直接拦住；第二层是候场区排队，有票的人一个个进场；第三层是检票闸机，拦住所有假票。

### Q2: Lua 脚本为什么能保证原子性？和分布式锁有什么区别？

**考察**：你是否理解 Redis 单线程模型

**A**：Redis 执行 Lua 脚本时，整段脚本是**原子执行**的——因为 Redis 是单线程模型，一段 Lua 脚本在执行期间不会被其他命令打断。

打个比方：Redis 就像一个只有一个窗口的银行，Lua 脚本就是一个客户的全套业务（查余额+扣款+开回执），窗口在处理你的时候不会突然切去给别人办。

和分布式锁的区别：

| 维度 | Lua 脚本 | 分布式锁（如 Redisson） |
|------|---------|---------------------|
| 原理 | Redis 单线程顺序执行 | 显式加锁 → 操作 → 解锁 |
| 性能 | 无锁等待开销，极快 | 有锁竞争，高并发下退化严重 |
| 复杂度 | 写一段 Lua 就行 | 锁续期、锁释放、防死锁一堆事 |
| 适用场景 | 逻辑简单、纯 Redis 操作 | 涉及多个外部资源的复杂操作 |

我的秒杀只需要在 Redis 内部做"判重 + 判库存 + 扣减"，全是 Redis 操作，Lua 完美匹配。如果要跨 Redis 和 MySQL 做原子操作，那就得上分布式锁或 TCC 事务了。

```lua
-- 我的 seckill.lua，一共就做了三件事：
local userExists = redis.call("EXISTS", KEYS[2])  -- 1. 你买过没？
if userExists == 1 then return -2 end              --    买过了滚

local stock = redis.call("GET", KEYS[1])           -- 2. 还有库存吗？
if tonumber(stock) <= 0 then return -1 end          --    没了滚

redis.call("DECR", KEYS[1])                        -- 3. 扣库存
redis.call("SETEX", KEYS[2], ARGV[1], "1")         --    标记你买过了
return tonumber(stock) - 1                          --    返回剩余库存
```

### Q3: Redis 扣完了但 DB 写失败怎么办？

**考察**：Redis-MySQL 双写一致性（这是你上次被问到的痛点！）

**A**：这个问题的本质是：**Redis 和 MySQL 不在同一个事务里**，Redis 先扣了库存，如果 DB 后面写失败了，库存就"凭空消失"了，实际的商品还在仓库但 Redis 说卖光了。

我的解决方案——**Consumer 的 catch 里做 Redis 回补**：

```java
// SeckillOrderConsumer.java 中的核心逻辑
try {
    doCreateOrder(seckillId, userId);  // 数据库操作
} catch (Exception e) {
    // 🚨 DB 失败了！必须回补 Redis！
    redisUtil.incr(seckillId);                      // 1. 库存加回来
    redisUtil.unmarkUserSeckilled(seckillId, userId); // 2. 清除用户标记
    log.error("❌ 秒杀订单创建失败，已回补 Redis");
}
```

**这个方案不完美**——如果"回补"本身也失败了呢（比如 Redis 那一刻挂了）？生产环境的终极兜底是**定时对账任务**：定期比对 Redis 库存和 DB 已卖数量，发现不一致就修正。

### Q4: MySQL 怎么兜底？`WHERE seckill_num > 0` 的原理是什么？

**考察**：数据库行锁原理

**A**：`UPDATE tb_newbee_mall_seckill SET seckill_num = seckill_num - 1 WHERE seckill_id = ? AND seckill_num > 0`

这条 SQL 的魔法在于 `seckill_num > 0`，它利用了 MySQL InnoDB 的**行级锁**：

1. 多个 Consumer 并发执行这条 UPDATE 时，InnoDB 会对这一行加**排他锁（X 锁）**
2. 其他线程必须排队等待
3. 第一个线程把 `seckill_num` 从 1 改成 0
4. 第二个线程拿到锁后，`seckill_num` 已经是 0，`WHERE seckill_num > 0` 不成立，`UPDATE` 影响行数 = 0
5. 代码检查 `updateCount <= 0` → 抛异常 → 事务回滚

**打个比方**：这就像超市最后一瓶牛奶，两个人同时伸手，但收银台一次只能结一个账——第一个人结完了，第二个人一看，货架上已经空了。

## 二、乐观锁与悲观锁

### Q5: 乐观锁和悲观锁的区别？

**考察**：并发控制的基本功

**A**：一句话区分——**悲观锁怕别人抢，先锁上再干活；乐观锁觉得没人抢，干完再检查有没有冲突**。

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 心态 | "肯定有人跟我抢" → 先锁了再说 | "应该没人跟我抢" → 先改了再说 |
| 实现方式 | `SELECT ... FOR UPDATE`、`synchronized` | 版本号（version）、CAS |
| 锁的时机 | **读的时候**就加锁 | **写的时候**才校验 |
| 适用场景 | 写多读少、竞争激烈 | 读多写少、竞争不激烈 |
| 性能 | 竞争激烈时还行，空闲时白白加锁浪费 | 竞争不激烈时快，激烈时大量重试反而慢 |

### Q6: 你项目里用了哪种？

**A**：我的 `WHERE seckill_num > 0` 本质上是**乐观锁思想**——不显式加锁，执行 UPDATE 时靠条件判断来检测冲突。

```
乐观锁的 CAS 味道：
  期望值：seckill_num > 0（我认为还有库存）
  执行：seckill_num = seckill_num - 1
  冲突检测：如果 UPDATE 影响行数 = 0，说明别人先改了 → 操作失败
```

如果用悲观锁，就要写 `SELECT * FROM tb_newbee_mall_seckill WHERE seckill_id = ? FOR UPDATE`，先锁住这行数据，再扣减。这在高并发下会导致大量线程阻塞等锁。

我选乐观锁的原因：秒杀场景下请求量巨大，但**经过 Redis + MQ 两层筛选后，真正到 MySQL 的并发量已经很小了**——乐观锁的"冲突少"前提成立。

### Q7: 乐观锁的 ABA 问题是什么？

**A**：经典面试追问。

ABA 问题：值从 A → 变成 B → 又变回 A，你的 CAS 检查"值还是 A"就以为没人动过。

打个比方：你把杯子放桌上去上厕所，回来看杯子还在，但其实你同事把你的水喝了又倒了一杯新的——杯子还是那个杯子，水已经不是那杯水了。

**解决方案**：加版本号。每次修改 version + 1，CAS 比较的不是值本身，而是 version。

```sql
-- 不加版本号（有 ABA 风险）
UPDATE goods SET stock = stock - 1 WHERE stock > 0

-- 加版本号（无 ABA 风险）
UPDATE goods SET stock = stock - 1, version = version + 1
WHERE id = ? AND version = ?
```

不过在我的秒杀场景里，**ABA 不是问题**——库存只会减不会加（除了回补场景），不存在"减了又加回来"的正常业务流。

## 三、Redis 核心知识

### Q8: Redis 到底会不会丢数据？

**考察**：Redis 持久化 + 主从复制机制（你上次把这个答成了缓存击穿，两个完全不同的概念！）

> 面试官问"Redis 丢数据"，这是**持久化和复制**的问题。和"缓存穿透/击穿/雪崩"毫无关系！不要混淆！

**A（满分体感答法）**：

**绝对会丢。** Redis 本质是内存数据库，它的持久化（RDB/AOF）和主从同步（Replication）**默认都是异步的**。

想象一个极端的物理场景：

```
时刻 1:  一笔秒杀扣减写入 Redis 主节点的内存     <- 数据只在内存里
时刻 2:  还没来得及刷入磁盘（RDB/AOF 还没触发）
         还没来得及通过网络同步给从节点（复制是异步的）
时刻 3:  主节点所在的物理机突然断电宕机  !!
时刻 4:  哨兵检测到主节点挂了 -> 把从节点选举为新主节点
结果:    这笔扣减数据在新主节点上 永久物理丢失 了
```

这不是理论推演——这是 Redis 官方文档明确说的 data safety 限制。所以**在绝对不能丢数据的场景里，不能单靠 Redis 扛最终一致性，必须有 MySQL 的事务做兜底**。

**然后再讲持久化方案**（降低丢数据的概率，但不能 100% 杜绝）：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **RDB**（快照） | 隔一段时间把整个内存 dump 成二进制文件 | 恢复快（直接加载文件）；文件紧凑 | 两次快照间的数据会丢（比如每 5 分钟一次，最多丢 5 分钟） |
| **AOF**（日志） | 每条写命令都追加到日志文件 | 数据更安全（`appendfsync always` 每条都刷盘，但性能差） | 文件大；恢复慢（要重放所有命令） |
| **混合持久化** | RDB 做底 + AOF 增量 | 兼顾速度和安全 | Redis 4.0+ 才支持 |

即使开了 AOF + `appendfsync everysec`（每秒刷盘），也最多丢 1 秒数据。如果用 `appendfsync always`（每条命令都刷盘），性能会暴跌，失去了用 Redis 的意义。

**所以结论是**：Redis 用来**扛并发、做预判**，但 MySQL 才是**数据真相的最终来源**。这正是我项目里 Redis Lua 预扣 + MySQL `WHERE seckill_num > 0` 兜底的设计原因。

**在你的秒杀项目里**：如果 Redis 挂了重启，秒杀库存数据丢失了——没关系！`RedisCacheHandlerUtil` 的 `CommandLineRunner` 会在项目启动时**自动从 MySQL 重新预热库存到 Redis**。相当于自带了一套"冷启动恢复机制"。

### Q9: 缓存穿透、击穿、雪崩 — 到底区别在哪？

**考察**：这三个是面试重灾区，很多人分不清

**A**：一张表彻底搞定——

| | 缓存穿透 | 缓存击穿 | 缓存雪崩 |
|---|---|---|---|
| **一句话** | 查一个**压根不存在**的数据 | **一个热点 key** 过期了 | **大量 key 同时**过期 |
| **比喻** | 你去图书馆借一本根本没出版的书，图书馆每次都要去仓库找 | 学校食堂只有一个窗口开，突然这个窗口关了 | 全校食堂所有窗口同时关了 |
| **流量走向** | 请求穿过缓存直接打穿到 DB | 一瞬间大量请求打到 DB | 海量请求同时打到 DB |
| **解决方案** | ① 布隆过滤器（先查有没有）<br>② 缓存空值（查不到也存个 null） | ① 热点 key 永不过期<br>② 互斥锁重建缓存 | ① key 的过期时间加随机值（打散）<br>② 多级缓存（本地 + Redis） |

**在你的秒杀项目里**：
- **不存在穿透问题**：因为秒杀商品是预热到 Redis 的，不存在的商品根本不会有秒杀 ID
- **不存在击穿问题**：库存 key 没有设过期时间（`set` 不带 TTL），不会自动过期
- 用户标记 key（`seckill:user:*`）带了过期时间，但它不是热点 key，过期也不影响

### Q10: Redis 有哪些数据结构？你项目里用了哪些？

**考察**：Redis 基础

**A**：五大基础类型 + 三种特殊类型：

| 类型 | 底层编码 | 啥时候用 | 你项目里用了吗 |
|------|---------|---------|--------------|
| **String** | SDS (Simple Dynamic String) | 计数器、缓存简单值 | ✅ 存库存 `seckill:stock:1 = "100"`<br>✅ 用户标记 `seckill:user:1:666 = "1"` |
| **List** | ziplist / quicklist | 消息队列、时间线 | ❌ 用了 RabbitMQ 替代 |
| **Hash** | ziplist / hashtable | 对象缓存（比存 JSON 省内存） | ❌ 没用到 |
| **Set** | intset / hashtable | 去重、交并差集 | ❌ 没用到 |
| **ZSet** | ziplist / skiplist（跳表） | 排行榜、延迟队列 | ❌ 没用到 |

**追问：ZSet 的跳表原理？为什么不用红黑树？**

跳表就像给有序链表加了"快速电梯"：

```
原始链表:  1 → 3 → 5 → 7 → 9 → 11 → 13
加一层:    1 ------→ 5 ------→ 9 -------→ 13
再加一层:  1 ---------------→ 9 -------→ 13
```

查找时从最高层开始，一路往右下跳，时间复杂度 O(log N)。

不用红黑树的原因（Redis 作者说的）：
1. **实现简单**——跳表几十行代码，红黑树几百行，bug 更少
2. **范围查询友好**——跳表底层就是有序链表，`ZRANGEBYSCORE` 直接遍历，红黑树要中序遍历
3. **内存灵活**——可以通过调整层数概率来 trade off 空间和时间

**追问：大 key / 热 key 怎么处理？**

| 问题 | 危害 | 解决方案 |
|------|------|---------|
| **大 key**（value 特别大） | 网络传输慢、删除时阻塞 | 拆分（大 Hash → 多个小 Hash）、`UNLINK` 异步删除 |
| **热 key**（QPS 特别高） | 单个 Redis 节点成瓶颈 | 本地缓存（Caffeine）兜底、key 打散到多个 slot |

### Q11: Redis 和数据库数据不一致怎么办？

**考察**：缓存一致性方案

**A**：这是个经典老大难问题。先说结论：**没有完美方案，只有在一致性和性能之间选一个平衡点**。

常见方案对比：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **先更新 DB，再删缓存**（Cache Aside） | 写 DB → 删 Redis | 主流方案，简单 | 删缓存和更新 DB 之间有窗口期 |
| **延迟双删** | 删 Redis → 写 DB → 等一会 → 再删 Redis | 缩小不一致窗口 | "等一会"的时间很难拿捏 |
| **Canal 监听 binlog** | DB 有变化自动同步 Redis | 解耦，强一致 | 引入额外组件，复杂度高 |

**你项目里的方案**：
你的秒杀不是典型的"缓存"场景——Redis 里的库存是**作为主数据使用的**（相当于 Redis 是主、DB 是备份）。所以你的一致性策略是：
1. **正向操作**：Redis 先扣 → MQ → DB 后扣（Redis 领先 DB）
2. **失败回补**：DB 失败时回补 Redis（保证 Redis 不会虚扣）
3. **启动预热**：重启时从 DB 重建 Redis（保证不丢数据）

## 四、异步下单（MQ 相关）

### Q12: 为什么不直接同步写数据库？

**考察**：为什么引入 MQ

**A**：打个比方——秒杀就像春运抢票。如果 1000 个人同时涌进售票窗口（MySQL），窗口直接被挤爆（连接池耗尽、CPU 100%）。

MQ 的作用就是在窗口前面拉了一条**排队线**：人还是那么多人，但变成一个个排队进窗口，窗口就不会被挤爆了。

具体好处：
1. **削峰填谷**——1000 QPS 的突发流量变成 Consumer 的 10 QPS 稳定消费
2. **解耦**——Service 层不需要知道订单怎么创建的，丢消息就走
3. **用户体验**——秒级返回"排队中"，不用等 DB 写完

---

### Q13: MQ 消息丢了怎么办？消费幂等怎么保证？

**考察**：MQ 可靠性

**A**：消息丢失可能发生在三个环节：

| 环节 | 怎么丢的 | 怎么防 |
|------|---------|--------|
| 生产者 → MQ | 网络抖了，没发出去 | 开启 **confirm 机制**，RabbitMQ 返回 ACK |
| MQ 自身 | RabbitMQ 重启了 | 队列和消息都设成**持久化**（durable + deliveryMode=2） |
| MQ → 消费者 | 消费到一半挂了 | 关闭自动 ACK，处理完了再**手动 ACK** |

消费幂等（同一条消息被重复消费不会出问题）：
- 方案 1：数据库唯一索引——`(seckill_id, user_id)` 联合唯一，插入重复直接报错
- 方案 2：Redis 判重——消费前检查 `seckill:user:1:666` 是否存在

我的项目里，**Redis 已经标记了用户**（Lua 脚本里的 `SETEX`），所以即使 MQ 重复消费，到数据库层面也会被唯一约束拦住。

### Q14: Consumer 里 `@Transactional` 为什么要单独抽方法？

**考察**：Spring AOP 代理原理（这是你代码里真实踩的坑！）

**A**：这是 Spring 最经典的坑之一——**同一个类内部的方法调用不走 AOP 代理**。

```java
@Component
public class SeckillOrderConsumer {

    @RabbitListener(...)
    public void processSeckillOrder(String message) {
        // 这里调用同一个类的方法 ↓
        doCreateOrder(seckillId, userId);  // ❌ @Transactional 不生效！
    }

    @Transactional  // ← 这个注解在同类调用时是摆设
    public void doCreateOrder(...) { ... }
}
```

**原因**：`@Transactional` 依赖 Spring AOP，AOP 通过**动态代理**包装你的 Bean。外部调用你的方法时走的是代理对象，但同一个类内部调用走的是 `this`（原始对象），绕过了代理。

打个比方：代理就像一个前台，外面来的客人都经过前台登记（事务管理），但你自己办公室里同事互相喊一声就过去了，不经过前台（没有事务）。

**解决方案**（二选一）：
1. **抽到独立 Service 类**——让 Spring 注入代理对象（生产推荐）
2. **注入自己** `@Autowired SeckillOrderConsumer self;` 然后 `self.doCreateOrder()`——强制走代理（不优雅但能用）

我代码里的注释也写了：*"生产环境建议将此方法抽到单独的 Service 类中"*。

## 五、性能相关

### Q15: 雪花算法原理？为什么不用 UUID 或自增 ID？

**考察**：分布式 ID 生成

**A**：雪花算法用一个 **64 位 long** 编码全局唯一 ID：

```
 0 | 0000...0000（41位时间戳）| 00000（5位机房）| 00000（5位机器）| 000000000000（12位序列号）
 ↑        可用约 69 年          0~31       0~31       同一毫秒最多 4096 个
符号位
```

生成过程（我的 `SnowflakeIdGenerator.nextId()`）：
1. 取当前时间戳（毫秒）
2. 同一毫秒内，序列号 +1（`& 4095` 防溢出）
3. 序列号满了就**自旋等下一毫秒**
4. 位运算拼起来：`(时间差 << 22) | (机房 << 17) | (机器 << 12) | 序列号`

为什么不用其他方案：

| 方案 | 问题 |
|------|------|
| UUID | 128 位字符串，太长；无序，插入 B+ 树时大量页分裂，索引性能差 |
| 数据库自增 | 强依赖 DB，高并发下成瓶颈；暴露业务量（ID=100 说明只卖了 100 单） |
| 雪花算法 | ✅ 64 位 long，紧凑；趋势递增，B+ 树友好；纯内存计算，不依赖外部 |

**追问：时钟回拨怎么办？**

如果服务器时间被手动往回调了，会导致生成重复 ID。我的代码里做了检测：

```java
if (timestamp < lastTimestamp) {
    throw new RuntimeException("时钟回拨！拒绝生成 ID");
}
```

更高级的方案：等待时间追上（小回拨），或者用 ZooKeeper 分配 workerId 避免 ID 碰撞（大回拨）。

### Q16: 库存预热怎么做的？

**考察**：项目工程细节

**A**：用 Spring Boot 的 `CommandLineRunner` 接口，项目**启动完成后自动执行**：

```java
@Component
public class RedisCacheHandlerUtil implements CommandLineRunner {
    @Override
    public void run(String... args) {
        // 1. 从 DB 查所有秒杀商品
        List<NewBeeMallSeckill> list = seckillMapper.findSeckillList();
        // 2. 遍历写入 Redis
        for (NewBeeMallSeckill seckill : list) {
            redisTemplate.opsForValue().set(
                "seckill:stock:" + seckill.getSeckillId(),
                seckill.getSeckillNum() + ""
            );
        }
    }
}
```

**追问：`CommandLineRunner` 在 Bean 生命周期的哪个阶段执行？**

在**所有 Bean 创建完毕 + `@PostConstruct` 执行完 + ApplicationContext 完全就绪之后**。是 Spring Boot 启动流程的最后一步，所以这时候 Redis 连接、MyBatis Mapper 都已经注入好了，可以放心使用。

### Q17: 如果 QPS 再涨 10 倍，你会怎么优化？

**考察**：架构扩展性思维

**A**：按瓶颈分层优化：

| 层级 | 瓶颈 | 优化方案 |
|------|------|---------|
| **前端** | 按钮被疯狂点击 | 按钮点一次后灰掉 5 秒（前端防刷） |
| **网关** | 请求太多 | 限流（令牌桶 / 滑动窗口），识别恶意 IP |
| **Redis** | 单节点扛不住 | Redis Cluster 分片；本地缓存（Caffeine）+ Redis 二级缓存 |
| **MQ** | 队列积压 | 增加 Consumer 数量；分区消费 |
| **MySQL** | 写入瓶颈 | 分库分表；异步批量插入 |

最关键的一步：在 Redis 前面再加一层**本地内存标记**（JVM 的 `ConcurrentHashMap`）——如果本地已经知道库存 = 0，连 Redis 都不用请求了，直接返回"卖光了"。

## 六、高并发四道防线 — 面试展开框架

> 面试官追问"如果量再大 10 倍怎么办"时，按这四道防线**自上而下**展开，体现系统化思维。
> 这些方案是对 Q17 的**深度拆解**，不再是一张表格式的罗列，而是一套有因果、有链路的叙事线。

---

### 防线一：外围阵地 — 风控拦截与动态路由（防线前置）

**核心痛点：** 黑产脚本绕过前端，利用海量动态 IP 直接轰炸下单接口。

**降维打法：**

**1. 动态 URL 隐藏**

绝不暴露真实下单接口。通过动态 Token 把秒杀入口变成"一次性暗门"：

```
Token = MD5(UserID + 商品ID + 时间戳 + 隐藏Salt)
```

- 前端在秒杀开始前一刻才拿到 Token，脚本没法提前抓接口
- 每个用户、每次请求的 Token 都不一样，批量脚本无法复用

**2. Redis 阅后即焚（防重放攻击）**

真实请求携带 Token 触达网关时：

```
网关收到 Token
  → Redis GET token:xxx  （验证是否存在）
  → 验证通过 → 立刻 DEL token:xxx  （阅后即焚）
  → 放行请求
```

Token 一旦被使用就立即销毁，同一个 Token **物理上不可能被第二次使用**，彻底杜绝重放攻击。

**和现有项目的关系：** 目前项目没有做风控层，直接暴露了秒杀接口。这是面试里可以主动提出的**升级方向**——"如果要防黑产，我会在网关前加一层动态 Token 校验"。

---

### 防线二：流量洪峰 — 热点拆分与极致抗压（单点破局）

**核心痛点：** 100 万并发集火单台 Redis 节点上的同一个商品 Key（比如 `seckill:stock:1`），单节点网卡和 CPU 瞬间雪崩。

> 这正是 Q10 提到的**热 key 问题**的终极解法，从"key 打散到多个 slot"展开。

**降维打法：**

**1. 库存分桶 (Bucket Sharding)**

将 100 件库存拆分到 10 个子 Key，利用 Redis Cluster 哈希槽自动打散到不同物理节点：

```
seckill:stock:1:bucket_0 = 10
seckill:stock:1:bucket_1 = 10
seckill:stock:1:bucket_2 = 10
...
seckill:stock:1:bucket_9 = 10
```

- 原来 1 个节点扛 100 万 TPS → 现在 10 个节点各扛 10 万 TPS
- TPS 随桶数**线性暴增**

**2. 无锁化跨桶借调**

用户请求进来，先通过哈希路由到固定桶（`hash(userId) % 10`）：

```java
int bucket = hash(userId) % BUCKET_COUNT;
long stock = redis.decr("seckill:stock:" + seckillId + ":bucket_" + bucket);
if (stock < 0) {
    // 命中空桶（局部售罄）→ 毫秒级轮询下一个桶
    redis.incr(...); // 回补当前桶
    bucket = (bucket + 1) % BUCKET_COUNT;
    // 重试下一个桶...
}
```

- 如果命中空桶（局部售罄），**在 Java 后端毫秒级轮询下一个桶**
- 绝不引入全局探查（比如遍历所有桶找最大值），避免读瓶颈

**3. 全局短路开关（OOS Flag）**

一旦所有桶的总库存扣空，立刻将全局 OOS (Out Of Stock) 标记推送到所有应用服务器的**本地 JVM 缓存**：

```java
// 和 Q17 提到的 ConcurrentHashMap 本地标记是同一个思路
private static final ConcurrentHashMap<Long, Boolean> LOCAL_OOS_MAP = new ConcurrentHashMap<>();

if (LOCAL_OOS_MAP.containsKey(seckillId)) {
    return "卖光了";  // 连 Redis 都不用请求，光速返回
}
```

后续海量流量在**服务器内存层被光速抛弃**，保护 Redis 物理集群。

**和现有项目的关系：** 目前项目是单 Key 存库存（`seckill:stock:1 = 100`），面试时可以说"当前方案在中等并发下够用，如果量级到百万级，我会做库存分桶 + JVM 本地短路"。

---

### 防线三：数据底线 — 异步解耦与最终一致性（交易闭环）

**核心痛点：** Redis 扣减成功，但发往 RabbitMQ 的消息因网络断开而丢失，导致超卖和数据不一致。

> 这是 Q3（Redis 扣完 DB 写失败）和 Q13（MQ 消息丢失）的**升级版**——不止要防 MQ → Consumer 的丢失，还要防 Producer → MQ 的丢失。

**降维打法：**

**1. 异步 ACK + 本地消息表**

扣减成功即返回前端"排队中"。同时在 Redis 或轻量级表暂存一条"发送中"的消息状态：

```
步骤 1: Redis Lua 扣减成功 → 返回前端
步骤 2: 写入本地消息表（状态 = SENDING）
步骤 3: 发送 MQ 消息
步骤 4: 收到 MQ 的 Publisher Confirms (ACK) → 清理消息表中该条记录
```

只有收到 MQ 的 ACK 才认为消息"真正送达"，否则消息留在暂存区等待重试。

**2. 看门狗兜底（定点微创）**

后台独立线程定期巡检消息表：

```java
@Scheduled(fixedRate = 5000)  // 每 5 秒扫一次
public void watchdog() {
    // 找出所有超时未 ACK 的消息（比如超过 10 秒还是 SENDING 状态）
    List<Message> failedMsgs = messageTable.findByStatusAndTimeout("SENDING", 10);
    for (Message msg : failedMsgs) {
        rabbitTemplate.send(msg);  // 重新投递
        msg.setRetryCount(msg.getRetryCount() + 1);
        if (msg.getRetryCount() > 3) {
            msg.setStatus("DEAD");  // 超过 3 次进死信，人工介入
        }
    }
}
```

- 只对**超时未 ACK 的极少数失败消息**进行重新投递
- **向前驱动系统**，绝不盲目全局回滚数据库

**3. 绝对幂等性（双保险）**

消费者端面对重发的重复消息，两层拦截：

| 层级 | 手段 | 特点 |
|------|------|------|
| **轻量前置** | Redis `SETNX seckill:dedup:{orderId}` | 毫秒级，挡掉 99% 重复 |
| **终极兜底** | MySQL 唯一复合索引 `(user_id, goods_id)` | 无锁，100% 可靠 |

彻底抛弃悲观锁，用**SETNX + 唯一索引**实现无锁幂等。

**和现有项目的关系：** 目前项目在 Q13 里依靠 RabbitMQ 自身的持久化 + 手动 ACK + 数据库唯一索引来保证。本地消息表 + 看门狗是**生产级升级方案**，面试时可以这样衔接——"我当前项目用的是 MQ 持久化 + 手动 ACK，如果要做到更高可靠性，我会引入本地消息表 + 看门狗巡检"。

---

### 防线四：战前准备 — 状态机与预热校验（安全点火）

**核心痛点：** 预热脚本因为网络抖动只写入了一半库存，带着残缺数据开启秒杀。

> 这是 Q16（库存预热）的**安全增强版**——不止要预热，还要**校验预热结果**。

**降维打法：**

**1. 活动状态机**

引入严格的秒杀活动状态流转，杜绝"带病上线"：

```
未开始 (INIT)
   ↓  触发预热
预热中 (WARMING)
   ↓  Checksum 校验通过
就绪 (READY)
   ↓  到达开始时间
进行中 (ACTIVE)
   ↓  库存归零或到达结束时间
已结束 (FINISHED)
```

- 只有 `READY` 状态才能接受用户请求
- `WARMING` 状态下即使到了开始时间也**不会切流**

**2. 预热幂等 + Checksum 数据对账**

预热写入必须幂等（重复执行不会写入双倍库存），写完后执行全量校验：

```java
// 预热完成后执行 Checksum
long redisTotal = 0;
for (int i = 0; i < BUCKET_COUNT; i++) {
    redisTotal += Long.parseLong(
        redis.get("seckill:stock:" + seckillId + ":bucket_" + i)
    );
}
long dbTotal = seckillMapper.getTotalStock(seckillId);

if (redisTotal == dbTotal) {
    // ✅ 100% 吻合 → 切换状态为 READY
    seckill.setStatus("READY");
} else {
    // ❌ 数据不一致 → 报警，不切流
    log.error("预热校验失败！Redis={}, DB={}", redisTotal, dbTotal);
    seckill.setStatus("WARMING");  // 保持预热状态，等待人工介入或自动重试
}
```

- 全量桶累加值 **必须 == 数据库总库存**，校验 100% 吻合后才切流为"就绪"状态
- 如果分桶场景下某个桶写入失败，Checksum 一定能发现

**和现有项目的关系：** 目前项目用 `CommandLineRunner` 在启动时直接预热，没有状态机也没有校验。面试时可以说"当前方案在单机部署下够用，如果是分布式部署 + 分桶库存，我会引入状态机 + Checksum 对账，确保预热数据完整才开放秒杀"。

### 面试展开话术（串联四道防线）

> 面试官："你的秒杀系统怎么应对高并发？"

**模板回答（30 秒版本）：**

我把高并发防护设计成四道防线，自上而下：

1. **最前面是风控层**——用动态 Token 隐藏真实接口，Redis 阅后即焚防重放，把黑产流量挡在门外
2. **第二层是热点拆分**——库存分桶打散到多个 Redis 节点，避免单点被打爆；库存清零后 JVM 本地标记短路，连 Redis 都不请求
3. **第三层是异步一致性**——Redis 扣减后通过 MQ 异步落库，本地消息表 + 看门狗保证消息不丢，SETNX + 唯一索引保证幂等
4. **最后是预热校验**——活动状态机控制流转，预热后 Checksum 对账，数据 100% 一致才开放秒杀

然后面试官追问哪一层，就往哪一层深入展开。
