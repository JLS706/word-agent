# 记忆系统和技能系统是怎么工作的？

## 记忆系统的整体设计是怎样的？

### 具体回答

`core/memory.py` 实现了**三级分层记忆**，学术上称为 **分级持久化记忆（Hierarchical Memory）**，设计目的是对抗大模型的 **灾难性遗忘（Catastrophic Forgetting）**——随着上下文变长，模型会逐渐"忘记"早期的关键指令。

| 层级 | 类型 | 存储 | 用途 |
|------|------|------|------|
| **L1 用户画像**（User Profile） | Markdown 画像（持续沉淀） | `memory/user_profile.md` | LLM 增量更新，注入 System Prompt 头部，内含铁律 section |
| **L2 经验记忆**（Experiential） | 向量记忆 (type=reflection) | `memory/memory_vectors.json` | 子任务反思提炼的经验，Embedding 向量化存入本地，Top-K 动态召回 |
| **L3 对话记忆**（Conversational） | 向量记忆 (type=conversation) | `memory/memory_vectors.json` | Q+A 摘要，30天 TTL + 召回率末尾淘汰 |

> 面试升级话术：*"为了对抗大模型的灾难性遗忘（Catastrophic Forgetting），我设计了分级持久化记忆（Hierarchical Memory）。最高优的 L1 不再是冰冷的宪法铁律列表，而是一份持续沉淀的 Markdown 用户画像——包含用户的写作风格偏好、导师要求、常用术语和绝对禁忌项。每次会话结束后由 LLM 增量更新，注入 System Prompt 头部，让 Agent 像一个'记得你所有习惯的老助理'一样工作；L2 的经验记忆通过 Embedding 向量化存入本地，使用时进行 Top-K 动态召回，彻底把大模型从长文本的负担中解放出来。"*

**记忆注入流程（静态-动态分离，Prompt Cache 友好）**：

```
用户发问 → Agent.run()
  ├── memory.recall_relevant(user_input)    # RAG 语义召回相关历史
  ├── skill_manager.match(user_input)       # 匹配相关技能手册
  └── _build_system_prompt(skills, memory)  # 拼装到消息序列
        ├── Message[0]: 静态根（L1 + 工具 + 行为准则）→ 可缓存
        └── Message[1]: 动态叶（技能 + 记忆）→ 每轮变化
```

## RAG 式向量记忆是怎么实现的？

### 具体回答

原理与文档 RAG 完全一样，只是数据源从"文档段落"变成了"对话摘要"：

**存入**（每轮对话结束后）：
```
Q + A 摘要 → EmbeddingClient.embed() → 向量 → VectorStore.add()
                                                → 持久化到 memory_vectors.json
```

**召回**（新对话开始时）：
```
用户问题 → EmbeddingClient.embed() → 查询向量
         → VectorStore.search(query_vec, top_k=3)
         → 余弦相似度排序 → 过滤低分 (< 0.45)
         → 格式化为 "[相关度 85.2%] 用户: ... Agent: ..."
```

**设计选择**：
- 自研 VectorStore（NumPy 实现），不依赖 Pinecone / Milvus 等外部向量数据库
- Q 和 A 拼在一起存储：`"用户: {input[:200]}\nAgent: {reply[:300]}"`，检索时语义更完整
- 失败时不影响主流程（`try/except` 包裹，只打 warning 日志）

## 用户偏好是怎么保存的？

### 具体回答

`Memory` 类提供 `set_preference(key, value)` 和 `get_preference(key)`，存储在 `history.json` 的 `preferences` 字段。

用途：比如用户习惯使用某种参考文献格式，Agent 可以记住这个偏好，下次自动应用。

## 记忆生命周期：双轨打分机制（RL-inspired）

记忆系统不仅有静态的分层，还引入了类似**强化学习（RL）**的双轨打分机制，管理一条记忆从**诞生**到**消亡**的完整生命周期。这是一套**自我纠错系统（Self-Correcting System）**。

### 第一轨：重要性分（Importance Score）——把控记忆的诞生（准入漏斗）

> 解决的问题：**"怎么写进来"**——降低写入成本，杜绝垃圾记忆。

对于未入库的在线日志，不依赖大模型盲目总结，而是引入**基于状态机流转的重要性积分**：

| 场景 | 重要性分 | 处理 |
|------|----------|------|
| 普通成功操作 | **0 分** | 直接丢弃 |
| 报错 → 拦截 → 换工具 → 成功自愈 | **极高** | 触发离线做梦，提炼为 L2 经验 |
| 人类强行接管（Human-in-the-loop） | **极高** | 同上 |

**核心思想**：只有 Agent 经历了艰难探索或被人类纠正的场景，才会通过准入漏斗，确保生出来的每一条记忆都是**极具信息熵的高质量避坑指南**。

### 第二轨：效用分（Utility Score）——决定记忆的存活（末位淘汰）

> 解决的问题：**"写错了/过时了怎么办"**——提高检索准确率，清除「毒记忆」。

一条经验入库 L2 后并非永远正确。随着外界环境变化（Word 版本升级、工具逻辑变更），旧经验可能变成导致幻觉的**「毒记忆」**。每条 L2 记忆都持有一个 `utility_score`（效用分）：

| 机制 | 触发条件 | 分值变化 | 说明 |
|------|----------|----------|------|
| **奖赏（Reward）** | 记忆被 RAG 召回 + Agent 凭借它成功完成工具调用 | **+0.1** | `reward_recalled_memories()` |
| **惩罚（Penalize）** | 记忆被 RAG 召回 + Agent 执行仍报错（误导） | **-0.1** | `penalize_recalled_memories()` |
| **时间衰减（Decay）** | 夜间做梦机制 | **×0.95** | 每晚全局衰减 |
| **淘汰（GC）** | `utility_score` 衰减至负数 | **物理抹除** | 反复报错 or 长期无人使用 |

**效果**：记忆系统不再是一个只进不出的垃圾桶，而是一个具有**「免疫淘汰机制」**的活体组织——高价值规律靠效用分不断强化存活；过时冗余信息在衰减与惩罚中被自动清洗。

### 架构师视角总结

```
重要性分（Importance Score）= 写入门控 → 降低写入成本
效用分  （Utility Score）  = 存活裁判 → 提高检索准确率

两轨合力 → 自我纠错系统（Self-Correcting Memory System）
```

> 面试升级话术：*"我的记忆系统不只是静态的三级分层，我还借鉴了强化学习的思路设计了双轨打分机制。第一轨是重要性分——只有 Agent 经历了报错→自愈的艰难探索，才能触发离线做梦提炼经验，保证写入质量；第二轨是效用分——每条 L2 记忆在实战中被召回后，成功加分、失败扣分，再叠加每晚的时间衰减，效用分降到负数就物理抹除。这样整个记忆系统就像一个有免疫淘汰能力的活体组织，不会积累过时的「毒记忆」。"*

## L1 改造：从宪法铁律到用户画像（User Profile）

### 为什么要改？

旧 L1 是 `learned_rules.json`，存储逐条铁律（如"必须关闭 Word"）。问题：
- **表达力弱**：JSON list 只能存"规则"，无法表达用户的写作风格、导师偏好、常用术语等软性画像
- **颗粒度粗**：每条规则都是"一票否决"级别，不适合存"用户倾向于用宋体"这类柔性偏好
- **维护成本高**：规则只能逐条增删，无法像文档一样结构化组织

### 改造方案

| 环节 | 旧方案 (JSON rules) | 新方案 (.md 画像) |
|------|---------------------|---------------------|
| **存储** | `memory/learned_rules.json` | `memory/user_profile.md` |
| **读取注入** | `load_rules_for_prompt()` 逐条拼装 | 直接读 `.md` 全文注入 System Prompt |
| **写入方式** | `SaveLearnedRuleTool`（三道门控，逐条追加） | `UpdateProfileTool`（LLM 读旧画像 + 新信息 → 重写整篇） |
| **三明治注入** | `build_l1_user_suffix()` 逐条拼接 | 从 `.md` 的 `## 铁律` section 提取摘要 |
| **后校验** | `_post_validate_l1()` 硬编码检查 | 保留，由画像中 `## 铁律` section 驱动 |

### 画像 `.md` 结构设计

```markdown
# 用户画像

## 基本信息
- 身份：XX大学硕士研究生
- 导师：张教授
- 研究方向：自然语言处理

## 写作偏好
- 正文字体：宋体小四，行距1.5倍
- 英文字体：Times New Roman
- 参考文献格式：GB/T 7714

## 铁律（绝对禁止项）
- 处理完文档后必须关闭 Word 进程
- 不得删除用户未明确要求删除的内容

## 历史踩坑
- 张教授的论文要求图表标题在图下方、表上方
- Word 2016 不支持 ContentControl，需用书签替代
```

### 技术要点

- **Prompt Cache 友好**：画像变更频率低（仅会话结束时更新），静态根前缀稳定，缓存命中率高
- **防膨胀**：画像有字数上限（如 1500 字），LLM 更新时强制精炼
- **铁律兼容**：`.md` 内 `## 铁律` section 保留旧 L1 的硬约束能力，三明治注入和后校验只读这个 section
- **向后兼容**：首次启动时，自动将旧 `learned_rules.json` 迁移为画像的 `## 铁律` section

> 面试话术：*"我后来把 L1 从冰冷的 JSON 规则列表升级为了一份 Markdown 用户画像。它不仅包含铁律禁忌，还涵盖了用户的写作风格、导师偏好、常用术语——让 Agent 像一个'记得你所有习惯的老助理'。每次会话结束后，LLM 读取旧画像加上新发现，增量重写整篇 profile，但有字数上限防膨胀。铁律 section 保留了旧架构的三明治注入和代码级后校验能力，做到了柔性画像和刚性约束的统一。"*

# Skill 技能系统

## Skill 系统解决什么问题？

### 具体回答

**问题**：System Prompt 的 Token 有限，不可能把所有业务知识都塞进去。

**解决方案**：Skill 系统实现了**按需加载的 Prompt 注入**：
- 每个 Skill 是一个 `.md` 文件，包含特定业务的操作手册
- 用户说"格式化参考文献"时，只加载参考文献相关的 Skill
- 用户说"你好"时，一个 Skill 也不加载（省 Token）

## Skill 的匹配策略是怎样的？

### 具体回答

`SkillManager.match()` 使用**双层匹配**：

**Layer 1：关键词匹配**（零 API 调用）
- 每个 Skill 文件的 frontmatter 定义了 `trigger_keywords`
- 用户输入包含任意关键词 → 匹配成功
- 按 `命中关键词数 × 优先级` 排序

**Layer 2：Embedding 语义匹配**（1 次 API 调用）
- 仅在 Layer 1 无命中时触发
- 预计算所有 Skill 的 Embedding（首次匹配时执行）
- 用余弦相似度找最相关的 Skill（阈值 0.5）

**设计思路与验证系统一致**：优先用免费的关键词匹配，只有边界情况才调 API。

## Skill 文件的格式是什么样的？

### 具体回答

```markdown
---
name: 参考文献格式化
description: 修复参考文献的字体、大小写、斜体等格式问题
trigger_keywords: [参考文献, 文献格式, reference, 格式化]
tools: [format_references]
priority: 10
---

## 操作步骤
1. 调用 format_references 工具
2. 参数说明：file_path = 文档路径
3. 常见问题：...
```

**frontmatter 不依赖 PyYAML**——用正则解析，保持零额外依赖。

## 新增业务需要改代码吗？

### 具体回答

不需要。只需在 `skills/` 目录下新增一个 `.md` 文件，启动时 `SkillManager._load_all()` 会自动扫描加载。**零代码，纯配置**。

# Prompt Cache 优化

## 为什么需要做 Prompt Cache 优化？

### 具体回答

**问题根源**：Transformer 底层没有物理隔离的“系统提示词”和“用户提示词”，它们都是展平拼接的一维 Token 数组。随着上下文变长，模型注意力被稀释，导致“中间注意力丢失”。

**解法**：大模型厂商提供了 Prompt Caching 机制——将 System Prompt 的 KV Cache 固化在显存中，多次请求共享。但这要求 Token 前缀完全一致。

## 旧架构有什么问题？

### 具体回答

旧架构将动态内容（skills、memory）嵌在 System Prompt 中间，导致每轮 System Prompt 都不一样，缓存永远命中不了。

## 新架构怎么解决的？

### 具体回答

**静态根 + 动态叶分离**：

```
Message[0]: SYSTEM（静态根）→ L1 规则 + 工具 + 行为准则 → 跨轮次不变 → 缓存命中
Message[1]: SYSTEM（动态叶）→ 技能 + 记忆 → 每轮变化，但不污染 [0] 的前缀
Message[2..n]: USER / ASSISTANT / TOOL → 正常对话
```

**效果**：在智谱 GLM / OpenAI 等支持隐式 Prompt Cache 的平台上，Message[0] 自动命中缓存，获得 50%~90% 输入 Token 降价。

**与 L1 防护网的关系**：
- 三明治注入（User 末尾）→ 不受影响
- 后置兆底校验 `_post_validate_l1()` → 不受影响
- L1 尾部重复（首尾夹击）→ 在静态根末尾，不受影响

### skill与长期记忆之争：

如果系统里只有 5 个 Skill（比如排版、摘要、查缩写），它们的向量在空间中离得非常远，匹配极其精准。

但如果你把所有稀奇古怪的“试错经验”全做成了 500 个 Skill 放进去。当用户输入一句模糊的指令时，必然会发生“语义碰撞（Semantic Collision）”。大模型会因为细微的相似度，错误地把“张导师的特殊表格规范”当成“李导师的图片规范”加载进来，这就是你说的“张冠李戴”。

### 面试官一定会追问：“什么错误是 Skill 搞不定，非要靠 Memory 的？”

举个具体的例子。用户在 Skill 里配置了‘将所有表格设置为自动调整列宽’。这是一个极其明确的确定性规则，放在 Skill 里完全没问题。
但在底层执行时，遇到了一种极端情况：某个表格跨页了，或者内部含有复杂的合并单元格。如果盲目调用 Word COM 的 AutoFit 接口，底层 DLL 会直接崩溃并抛出极其晦涩的 pywintypes.com_error。这个底层的异常，靠表层的 Skill 是防不住的。
这时，长时记忆（Memory）的价值就体现了。由于之前系统在这个坑里摔过，Agent 曾通过 ReAct 的反复试错，得出了一套经验：‘遇到跨页表格报错时，不能对整个 Table 调 AutoFit，必须先 catch 异常，然后降级为遍历 Cell 逐行调整’。系统将这个‘报错特征’与‘成功绕过的动作序列’向量化存入记忆库。
下一次，当 Executor 在另一篇文档遇到同样的合并表格报错时，它会立刻从记忆库中召回这套启发式的避坑指南，瞬间完成自我修复。Skill 决定了系统能飞多高，而基于错误特征召回的 Memory 决定了系统摔倒后能不能自己爬起来。”
