# 记忆系统和技能系统是怎么工作的？

## 记忆系统的整体设计是怎样的？

### 具体回答

`core/memory.py` 实现了**三级分层记忆**，学术上称为 **分级持久化记忆（Hierarchical Memory）**，设计目的是对抗大模型的 **灾难性遗忘（Catastrophic Forgetting）**——随着上下文变长，模型会逐渐"忘记"早期的关键指令。

| 层级 | 类型 | 存储 | 用途 |
|------|------|------|------|
| **L1 宪法记忆**（Constitutional） | 铁律（永不过期） | `memory/learned_rules.json` | 硬编码在 System Prompt 头部，三明治注入 + 后校验 |
| **L2 经验记忆**（Experiential） | 向量记忆 (type=reflection) | `memory/memory_vectors.json` | 子任务反思提炼的经验，Embedding 向量化存入本地，Top-K 动态召回 |
| **L3 对话记忆**（Conversational） | 向量记忆 (type=conversation) | `memory/memory_vectors.json` | Q+A 摘要，30天 TTL + 召回率末尾淘汰 |

> 面试升级话术：*"为了对抗大模型的灾难性遗忘（Catastrophic Forgetting），我设计了分级持久化记忆（Hierarchical Memory）。最高优的 L1 宪法记忆硬编码在 System Prompt 头部，代表系统铁律（如'必须关闭 Word 进程'）；L2 的经验记忆通过 Embedding 向量化存入本地，使用时进行 Top-K 动态召回，彻底把大模型从长文本的负担中解放出来。"*

**记忆注入流程（静态-动态分离，Prompt Cache 友好）**：

```
用户发问 → Agent.run()
  ├── memory.recall_relevant(user_input)    # RAG 语义召回相关历史
  ├── skill_manager.match(user_input)       # 匹配相关技能手册
  └── _build_system_prompt(skills, memory)  # 拼装到消息序列
        ├── Message[0]: 静态根（L1 + 工具 + 行为准则）→ 可缓存
        └── Message[1]: 动态叶（技能 + 记忆）→ 每轮变化
```

## RAG 式向量记忆是怎么实现的？

### 具体回答

原理与文档 RAG 完全一样，只是数据源从"文档段落"变成了"对话摘要"：

**存入**（每轮对话结束后）：
```
Q + A 摘要 → EmbeddingClient.embed() → 向量 → VectorStore.add()
                                                → 持久化到 memory_vectors.json
```

**召回**（新对话开始时）：
```
用户问题 → EmbeddingClient.embed() → 查询向量
         → VectorStore.search(query_vec, top_k=3)
         → 余弦相似度排序 → 过滤低分 (< 0.45)
         → 格式化为 "[相关度 85.2%] 用户: ... Agent: ..."
```

**设计选择**：
- 自研 VectorStore（NumPy 实现），不依赖 Pinecone / Milvus 等外部向量数据库
- Q 和 A 拼在一起存储：`"用户: {input[:200]}\nAgent: {reply[:300]}"`，检索时语义更完整
- 失败时不影响主流程（`try/except` 包裹，只打 warning 日志）

## 用户偏好是怎么保存的？

### 具体回答

`Memory` 类提供 `set_preference(key, value)` 和 `get_preference(key)`，存储在 `history.json` 的 `preferences` 字段。

用途：比如用户习惯使用某种参考文献格式，Agent 可以记住这个偏好，下次自动应用。

# Skill 技能系统

## Skill 系统解决什么问题？

### 具体回答

**问题**：System Prompt 的 Token 有限，不可能把所有业务知识都塞进去。

**解决方案**：Skill 系统实现了**按需加载的 Prompt 注入**：
- 每个 Skill 是一个 `.md` 文件，包含特定业务的操作手册
- 用户说"格式化参考文献"时，只加载参考文献相关的 Skill
- 用户说"你好"时，一个 Skill 也不加载（省 Token）

## Skill 的匹配策略是怎样的？

### 具体回答

`SkillManager.match()` 使用**双层匹配**：

**Layer 1：关键词匹配**（零 API 调用）
- 每个 Skill 文件的 frontmatter 定义了 `trigger_keywords`
- 用户输入包含任意关键词 → 匹配成功
- 按 `命中关键词数 × 优先级` 排序

**Layer 2：Embedding 语义匹配**（1 次 API 调用）
- 仅在 Layer 1 无命中时触发
- 预计算所有 Skill 的 Embedding（首次匹配时执行）
- 用余弦相似度找最相关的 Skill（阈值 0.5）

**设计思路与验证系统一致**：优先用免费的关键词匹配，只有边界情况才调 API。

## Skill 文件的格式是什么样的？

### 具体回答

```markdown
---
name: 参考文献格式化
description: 修复参考文献的字体、大小写、斜体等格式问题
trigger_keywords: [参考文献, 文献格式, reference, 格式化]
tools: [format_references]
priority: 10
---

## 操作步骤
1. 调用 format_references 工具
2. 参数说明：file_path = 文档路径
3. 常见问题：...
```

**frontmatter 不依赖 PyYAML**——用正则解析，保持零额外依赖。

## 新增业务需要改代码吗？

### 具体回答

不需要。只需在 `skills/` 目录下新增一个 `.md` 文件，启动时 `SkillManager._load_all()` 会自动扫描加载。**零代码，纯配置**。

# Prompt Cache 优化

## 为什么需要做 Prompt Cache 优化？

### 具体回答

**问题根源**：Transformer 底层没有物理隔离的“系统提示词”和“用户提示词”，它们都是展平拼接的一维 Token 数组。随着上下文变长，模型注意力被稀释，导致“中间注意力丢失”。

**解法**：大模型厂商提供了 Prompt Caching 机制——将 System Prompt 的 KV Cache 固化在显存中，多次请求共享。但这要求 Token 前缀完全一致。

## 旧架构有什么问题？

### 具体回答

旧架构将动态内容（skills、memory）嵌在 System Prompt 中间，导致每轮 System Prompt 都不一样，缓存永远命中不了。

## 新架构怎么解决的？

### 具体回答

**静态根 + 动态叶分离**：

```
Message[0]: SYSTEM（静态根）→ L1 规则 + 工具 + 行为准则 → 跨轮次不变 → 缓存命中
Message[1]: SYSTEM（动态叶）→ 技能 + 记忆 → 每轮变化，但不污染 [0] 的前缀
Message[2..n]: USER / ASSISTANT / TOOL → 正常对话
```

**效果**：在智谱 GLM / OpenAI 等支持隐式 Prompt Cache 的平台上，Message[0] 自动命中缓存，获得 50%~90% 输入 Token 降价。

**与 L1 防护网的关系**：
- 三明治注入（User 末尾）→ 不受影响
- 后置兆底校验 `_post_validate_l1()` → 不受影响
- L1 尾部重复（首尾夹击）→ 在静态根末尾，不受影响

**skill与长期记忆之争**：
如果系统里只有 5 个 Skill（比如排版、摘要、查缩写），它们的向量在空间中离得非常远，匹配极其精准。

但如果你把所有稀奇古怪的“试错经验”全做成了 500 个 Skill 放进去。当用户输入一句模糊的指令时，必然会发生“语义碰撞（Semantic Collision）”。大模型会因为细微的相似度，错误地把“张导师的特殊表格规范”当成“李导师的图片规范”加载进来，这就是你说的“张冠李戴”。

更可怕的是，Skill 是一旦命中就全量注入 System Prompt 的。错误加载不仅耗费 Token，还会给大模型下达相互矛盾的强指令，导致系统当场精神分裂。

2. L2 记忆的绝妙定位：长尾的“柔性补丁”
正如你所说，L2 存在的意义，就是接管那 80% 的“不那么标准化、极度罕见、甚至只发生过一次”的异常分支。

Skill 是“硬路由（Hard Routing）”：它是必须严格遵守的 SOP（标准作业程序）。

L2 是“软提示（Soft Hint）”：它是 RAG 召回的参考资料。在 agent.py 构建动态上下文时，L2 的经验是作为 [历史记忆] 附加进去的。大模型看到它时，会把它当成一种“通关攻略”或“前人踩坑记录”来参考，而不是强制覆盖系统设定的底线。
