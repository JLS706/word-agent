# 错误处理和容错机制是怎么设计的？

## Agent 遇到工具执行错误时会怎么处理？

### 具体回答

`core/agent.py` 实现了**三级错误分类 + 结构化恢复引导**：

```
工具执行失败 → _classify_error() 三级分类
  ├── Level 1 (transient): 临时性错误 → 自动重试（指数退避 2s→4s→8s）
  ├── Level 2 (correctable): LLM 可自修正 → 构造结构化 Observation 引导 LLM
  └── Level 3 (fatal): 不可恢复 → 告知 LLM 无法解决，建议通知用户
```

## 三级错误分类具体是怎么判断的？

### 具体回答

`_classify_error()` 方法基于**错误文本模式匹配 + 异常类型**：

**Level 1 - 临时性错误**（可自动重试，不消耗 LLM 推理步骤）：
- 关键词：`超时, timeout, RPC, 网络, 429, rate limit, -2147`（COM 错误码）
- 策略：指数退避重试（2, 4, 8 秒），最多 3 次

**Level 2 - LLM 可自修正的错误**（需要 LLM 调整参数或策略）：
- 2a. 文件路径问题：`FileNotFoundError`, `不存在, 找不到`
- 2b. 参数类型问题：`TypeError, KeyError`
- 2c. Word COM 接口问题：`COM, com_error, Word`
- 2d. 权限问题：`PermissionError, 权限`
- 策略：每类错误提供具体的修正建议列表

**Level 3 - 不可恢复的错误**：
- 不匹配任何已知模式的兜底
- 建议 LLM 向用户说明情况

## 结构化 Observation 是什么意思？

### 具体回答

传统做法是直接把 traceback 传给 LLM，但 traceback 很长、噪音多、浪费 Token。

`_build_error_observation()` 构造的结构化 Observation 包含：

```
⚠️ 工具 format_references 执行失败
错误分类: 文件路径错误 (FileNotFoundError)
错误详情: [截断到 200 字符]
调用参数: file_path='C:\...\test.docx'

修正建议:
  1. 检查文件路径拼写是否正确
  2. 调用 recall_history 查看上次处理过的文件路径
  3. 向用户询问正确的文件路径

📌 当前尝试 1/3 次。请先分析失败原因，修正参数后重新调用。
```

**设计优势**：
1. **省 Token**——不传完整 traceback，只传关键信息
2. **引导修正方向**——给 LLM 具体的修正建议，而不是让它自己猜
3. **防止无限循环**——显示当前重试次数，超过上限时明确告知"不要再重复相同调用"

## 临时性错误的自动重试是怎么实现的？

### 具体回答

在 `_execute_tool()` 中：

```python
if level == "transient" and attempt < max_attempts:
    wait_sec = 2 ** attempt    # 指数退避: 2s, 4s, 8s
    time.sleep(wait_sec)
    return self._execute_tool(call_id, name, arguments)  # 递归重试
```

关键点：
- **指数退避**：给外部服务恢复时间（API 限流、Word 进程繁忙等）
- **不消耗 LLM 步骤**：重试在 Agent 内部完成，LLM 不感知
- **透明恢复**：如果重试成功，LLM 只看到成功结果

## Checkpoint 断点续传是怎么设计的？

### 具体回答

`core/checkpoint.py` 实现了**游戏存档式**的断点续传：

**核心类**：
- `WorkflowPhase` (Enum): 7 个阶段状态 `NOT_STARTED → PLANNING → PLAN_DONE → EXECUTING → EXEC_DONE → REVIEWING → COMPLETED`
- `WorkflowState`: 完整的状态快照（阶段、计划、已完成步骤、重试计数等）
- `Checkpointer`: 存档管理器（save / load / clear / list）

**存储格式**：JSON 文件，存放在 `checkpoints/ckpt_{task_id}.json`

**恢复逻辑**：
```python
state = checkpointer.load(task_id)
if state.phase in (PLAN_DONE, EXECUTING):
    # 跳过 Phase 1，直接进入 Phase 2
    # 并从 state.current_step_index 恢复执行位置
```

**为什么用 JSON 而不是 SQLite / Redis？**
- 简单可靠，便于调试（直接打开文件就能看状态）
- 单用户单任务场景，不需要并发写入
- 文件数量有限（每个任务只有一个 checkpoint 文件）
