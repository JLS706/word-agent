# 这个项目是做什么的？整体架构是怎样的？

## 用一句话介绍一下这个项目

### 具体回答

DocMaster Agent 是一个**学术论文排版 AI 智能助手**，通过自然语言指令驱动 Word 文档自动化处理。基于 ReAct (Reasoning + Acting) 架构，参考 OpenManus 和 smolagents 的设计理念打造。

核心价值：用户说一句"帮我格式化参考文献并生成交叉引用"，Agent 就能自动拆解任务、调用工具、验证结果。

## 为什么选择 ReAct 架构而不是其他方案（如 Chain-of-Thought、Plan-and-Execute）？

### 具体回答

**ReAct 的核心优势是"边想边做"**：

| 架构 | 特点 | 适用场景 |
|------|------|----------|
| CoT | 只推理，不与环境交互 | 数学推理、知识问答 |
| Plan-and-Execute | 先制定完整计划，再一次性执行 | 确定性任务 |
| **ReAct** | **Think → Act → Observe 循环** | **需要与外部工具交互、结果不确定的任务** |

选 ReAct 的原因：
1. **文档处理结果不确定**——格式化后可能有遗漏，需要观察结果再决定下一步
2. **支持动态调整**——第一步失败了可以改参数重试，不需要重新规划整个计划
3. **天然兼容 Function Calling**——LLM 返回 tool_calls 就是 Act，返回文本就是 Finish

代码体现：`core/agent.py` 的 `run()` 方法就是一个标准的 ReAct 循环：
```
for step in range(max_steps):
    response = llm.chat(history, tools)    # Think
    if not response.tool_calls:            # 无工具调用 → Finish
        return response.content
    for tc in response.tool_calls:         # Act
        result = execute_tool(tc)          # Observe（结果加入 history）
```

## 项目的整体架构分层是怎样的？

### 具体回答

```
┌─────────────────────────────────────┐
│  入口层: main.py / api.py (FastAPI) │  ← 用户交互
├─────────────────────────────────────┤
│  编排层: core/                       │
│  ├── agent.py     (单Agent ReAct)   │  ← 核心推理引擎
│  ├── multi_agent.py (多Agent流水线)  │  ← Planner→Executor→Reviewer
│  ├── checkpoint.py (断点续传)        │  ← 状态持久化
│  ├── memory.py    (记忆系统)         │  ← 跨会话记忆+RAG
│  └── skills.py    (技能加载器)       │  ← 动态Prompt注入
├─────────────────────────────────────┤
│  工具层: tools/                      │
│  ├── base.py      (Tool基类+注册表) │  ← 插件化接口
│  ├── ref_*.py, fig_*.py ...         │  ← 文档处理工具
│  ├── tool_creator.py (动态工具创建) │  ← Agent自己写工具
│  └── custom/      (用户自定义工具)   │  ← 运行时扩展
├─────────────────────────────────────┤
│  安全层: core/sandbox.py             │
│  ├── Layer 1: AST 静态分析          │
│  ├── Layer 2: builtins 白名单        │
│  └── Layer 3: 进程隔离 + 超时强杀    │
├─────────────────────────────────────┤
│  基础设施: LLM / Embeddings / Docker │  ← OpenAI兼容接口
└─────────────────────────────────────┘
```

## 技术栈是怎样的？

### 具体回答

- **语言**: Python 3.10+
- **LLM 接口**: OpenAI SDK（兼容 Gemini / 智谱 / 硅基流动等）
- **文档处理**: python-docx + Word COM 自动化 (pywin32)
- **Web 框架**: FastAPI（API 模式）
- **容器化**: Docker + Docker Compose（Agent 容器 + Sandbox 微服务）
- **向量检索**: 自研轻量 VectorStore（NumPy 余弦相似度，无外部向量数据库依赖）
- **日志**: Python logging 模块（统一日志系统）

## 这个项目和用 LangChain / LangGraph 有什么区别？

### 具体回答

DocMaster 是**从零手写**的轻量级 Agent 框架，没有使用 LangChain：

| 对比维度 | LangChain/LangGraph | DocMaster |
|----------|---------------------|-----------|
| 依赖量 | 几十个包 | 仅 openai + python-docx + pywin32 |
| 学习成本 | 高（大量抽象概念） | 低（代码即文档，每个模块 100-300 行） |
| 灵活性 | 受框架约束 | 完全可控，按需定制 |
| Agent 循环 | 框架封装 | `agent.py` 手写 ReAct 循环 (~100行) |
| 记忆系统 | 需要集成外部向量DB | 自研 VectorStore + JSON 持久化 |
| Checkpoint | LangGraph 专有 | 自研 JSON 存档系统 |

**核心观点**：对于特定领域的 Agent（如本项目只做学术论文排版），手写框架更轻量、更可控。LangChain 适合快速原型，但在生产环境中往往需要深度定制。
