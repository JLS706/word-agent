# 简历模板 — STAR 法则

> **STAR 法则**：Situation（背景）→ Task（任务）→ Action（行动）→ Result（结果）
>
> 以下提供 2 个简历版本，根据应聘岗位选择。

---

## 版本 A：AI 工程师 / LLM 应用方向

### 项目名称
**DocMaster Agent — 基于 Multi-Agent 架构的学术论文排版智能助手**

### 项目描述（控制在 3 行内）
基于 ReAct (Reasoning + Acting) 架构，从零设计并实现的轻量级 AI Agent 系统。通过自然语言指令驱动 Word 文档自动化处理，支持参考文献格式化、交叉引用生成、格式诊断等 10+ 工具。实现了 Multi-Agent 三角色协作（Planner→Executor→Reviewer）、三级分层记忆、语义切块 RAG、Skill 热插拔、Checkpoint 断点续传等企业级特性。

### 核心贡献（STAR 法则，每条 2-3 行）

**1. 设计并实现 ReAct 推理引擎**
- **S/T**：需要让 LLM 自主规划并执行多步文档操作任务，且能处理执行中的不确定性
- **A**：手写 Think→Act→Observe 循环（~100 行），集成 OpenAI Function Calling 实现工具自主调用；设计三级错误分类（临时/可修正/致命）+ 结构化 Observation 引导 LLM 自修正
- **R**：Agent 首轮工具调用成功率提升至 ~90%，错误自修正覆盖率 >70%

**2. 构建 Multi-Agent 三角色协作流水线**
- **S/T**：单 Agent 处理复杂任务（全面排版一篇论文）时，计划与执行混杂、缺乏验证
- **A**：设计 Planner→Executor→Reviewer Pipeline；实现逐步执行 + 两级验证（规则匹配 + LLM mini-review）+ 三级回溯修正（重试→重规划→跳过）；引入 Checkpoint 断点续传保障长任务可靠性
- **R**：Pipeline 全流程自动化完成率 >85%，中途崩溃可从断点恢复

**3. 设计三级分层记忆系统**
- **S/T**：Agent 需要跨会话学习经验，同时保证核心规则不被 LLM 违反
- **A**：设计 L1 核心规则（铁律，三明治注入 + 后校验）/ L2 长期反思（向量 RAG，冲突融合）/ L3 短期对话（30 天 TTL + 召回率淘汰）三层架构；实现记忆晋升（L3→L2）和效用衰减机制
- **R**：L1 规则服从率 >95%，L2 经验召回准确率 >80%

**4. 实现 Prompt Cache 优化，降低 API 成本 50%+**
- **S/T**：System Prompt 包含动态内容（技能、记忆），导致厂商 KV Cache 永远命中不了
- **A**：将 System Prompt 拆为静态根（L1+工具+行为准则，跨轮次不变）+ 动态叶（技能+记忆，独立消息），确保前缀不变以命中 Prompt Cache；配合动态宪法挂载精选 2-3 条相关规则注入
- **R**：输入 Token 成本降低 50%~90%（取决于厂商 Cache 策略）

**5. 构建四层纵深防御安全沙盒**
- **S/T**：Agent 具备代码执行能力（execute_python），需防止 LLM 生成的代码造成安全风险
- **A**：设计 AST 静态分析 → builtins 白名单 → 进程隔离+超时强杀 → Docker 容器隔离四层防护；实现严格模式（只读分析）和工具模式（允许文件操作）两种安全级别
- **R**：零安全事故，支持 Agent 在沙盒中自主创建和测试新工具

### 技术栈
Python 3.10+ / OpenAI Function Calling / ReAct Agent / Multi-Agent Pipeline / RAG / Embedding / NumPy / FastAPI / Docker / pywin32 (Word COM)

## 版本 B：后端工程师 / 基础架构方向

### 项目名称
**DocMaster Agent — 多智能体文档处理系统**

### 项目描述
设计并实现多智能体协作系统，整合 LLM 推理引擎、插件化工具框架、分层记忆系统、安全沙盒等模块。支持通过 RESTful API 接收自然语言指令，自动化处理 Word 文档。

### 核心贡献

**1. 设计插件化工具框架**
- **S/T**：系统需支持 10+ 工具且允许运行时动态扩展
- **A**：实现 Tool 抽象基类 + ToolRegistry 注册表模式；支持 Agent 运行时创建新工具（代码生成→沙盒验证→用户审批→热注册）；Skill 插件系统支持零代码新增业务（.md 文件热加载）
- **R**：新增工具只需继承基类 + 实现 execute()，零改 Agent 核心代码

**2. 实现状态机驱动的工作流引擎**
- **S/T**：多步骤文档处理流水线需要支持错误恢复和断点续传
- **A**：设计 9 阶段状态机（WorkflowPhase）；实现三级回溯策略（重试→重规划→跳过）；JSON 文件 Checkpoint 持久化，崩溃后可从任意节点恢复
- **R**：流水线中断恢复成功率 100%

**3. 构建向量检索系统（自研，零外部依赖）**
- **S/T**：需要语义检索能力，但不想引入 Pinecone / Milvus 等重型依赖
- **A**：基于 NumPy 实现余弦相似度计算 + VectorStore（JSON 缓存持久化）；设计语义切块算法（Embedding 相似度断崖检测，自适应阈值 μ-1σ）
- **R**：检索延迟 <100ms（千条级），零外部数据库依赖

**4. 设计多层安全架构**
- **S/T**：系统具备代码执行能力，需要纵深防御
- **A**：4 层安全架构（AST→builtins→进程隔离→Docker）；COM 进程守护（PID 精准狙击 + 残留清理）；文档事务（备份→执行→回滚/提交）
- **R**：零安全事故、零进程泄漏

### 技术栈
Python / FastAPI / Docker / Docker Compose / NumPy / pywin32 / OpenAI SDK / JSON 持久化

## 面试口述模板（3 分钟版）— SCQA 泥坑叙事框架

> 面试官问："介绍一下你的项目？"
>
> **核心原则**：不要上帝视角背架构名词，要泥坑视角讲踩坑故事。
> 用极度具象的场景把面试官拉进来，让他"感同身受"。

---

### 第一段：定调 — 把面试官拉进真实业务的泥坑（~40 秒）

> **叙事功能**：Situation（情境）+ Complication（冲突），建立共鸣

**话术**：

"面试官您好，我的核心项目是一个学术论文排版 AI 智能助手。目前市面上大部分 Agent 都在做云端的文档问答，但我的项目是直接扎进本地，通过底层 COM 接口操控 Word。因为我发现，处理论文时最折磨人的根本不是总结摘要，而是那些极度繁琐的格式：比如几十条参考文献的大小写统一、成百上千个图注的自动编号和交叉引用。这是一个极度缺乏容错率的'重泥沼'业务。"

---

### 第二段：用一个极度具象的例子替换干瘪名词（~80 秒）

> **叙事功能**：Action（行动），用"100 页烂文档"的完整流转交代 Multi-Agent 协作

**话术**：

"为了解决这个问题，我发现单 Agent 根本搞不定，它改到一半就会忘记上下文，甚至把文档改乱。所以我引入了 Multi-Agent 协作流。

我给您举个真实的例子：假设用户丢进来一份 100 页、完全没排版的烂文档。

系统启动后，**Planner 绝对不会直接动手**，它会先通过 `analyze_document` 工具扫全篇，发现有 50 个手写图注和 30 条文献，然后生成一个严格的顺序计划。

接着，**Executor 拿着计划一步步调接口去改**。比如在改图注的交叉引用时，它不是盲目替换，而是通过我设计的'**动态锚点防护盾**'去精准挂载书签。

最后，**Reviewer 会去验收**。它不会只听 Executor 报告成功，而是亲自去读取底层的域代码。如果发现'错误！未找到引用源'这种交叉引用断裂，Reviewer 会直接打回，触发我底层写的**三级回溯熔断机制**，尝试把原书签转成纯文本或者退避重试。"

---

### 第三段：抛出底层硬核工程挑战，完成降维打击（~60 秒）

> **叙事功能**：Result（结果）+ Flex（亮肌肉），用底层工程细节拉开差距

**话术**：

"在这个项目的落地过程中，我发现最大的挑战根本不是如何调用大模型，而是**系统底层的稳定性和算力瓶颈**。

比如，为了防止大模型生成的恶意指令搞挂服务器，我没有用简单的正则拦截，而是上了 **Docker + Windows 进程守护的双轨沙盒**；

为了防止 Word 这种笨重进程卡死吃光内存，我抛弃了暴力的超时 Kill，自研了 **COMSafeLock**，利用 **PID 差集法**进行精准的僵尸进程查杀和快照回滚；

面对 100G 海量文本的相似度计算，我没有无脑请求云端导致被限流，而是利用**本地 24M 的轻量级模型做动态批处理**算 Embedding。

目前，这套系统已经能在一个极其受限的单机内存环境下，全自动、稳定地把一篇混沌的纯文本结构化成符合学术规范的 Word 文档。"

## Java 八股翻盘路由 — 弱势话题桥接到 Agent 强项

> **使用场景**：面试官问 Java 基础八股你答不深时，不要慌不要硬撑，用以下桥接话术把话题**自然地引向你最强的 Agent 项目**。
>
> **核心心法**：先正面回答 Java 基础（不能跑题），然后用"在我的 Agent 项目里遇到了一个更极端的场景……"自然过渡。

### 翻盘路由 ①：被问"并发与锁" → 引出 COMSafeLock

**触发场景**：面试官问乐观锁/悲观锁、synchronized、CAS、线程安全

**桥接话术**：

"在传统的秒杀系统里，我们常讨论乐观锁和悲观锁去防超卖。但在做 DocMaster Agent 时，我遇到了一个比数据库并发更棘手的'锁'问题——**Windows 底层 COM 进程的线程安全与死锁**。

大模型生成的代码具有极强的不确定性，如果直接让它去操作 Word，一旦大模型输出死循环，整个 Word COM 接口就会彻底死锁，甚至把宿主机的内存吃干。为了解决这个并发与隔离问题，我没有用简单的 Java 线程锁，而是自研了 **COMSafeLock 上下文管理器**。我通过 `DispatchEx` 强制隔离出独立的子进程，并利用 **PID 差集法**做精准的并发追踪。这就相当于在操作系统层面，为大模型的执行加了一把带有'超时强杀与事务回滚'功能的**物理级悲观锁**。"

📖 深挖弹药：[04-安全沙盒系统](./04-安全沙盒系统.md) + [10-面试QA](./10-面试QA.md) Q21

---

### 翻盘路由 ②：被问"Redis / MQ / 缓存" → 引出背压引擎

**触发场景**：面试官问 Redis 高可用、MQ 消息丢失、分布式缓存

**桥接话术**：

"我之前确实接触过使用 Redis 和 MQ 来做流量削峰的分布式架构。但在深入研发 AI Agent 后，我对这种中间件的过度依赖有了反思。

在处理超长文档的'**语义切块（Semantic Chunking）**'时，系统瞬间会产生数万个文本碎块。如果还按照传统后端的思维，把这些碎块扔进 Redis 或 Kafka，跨进程的序列化和网络 I/O 会导致成百上千倍的延迟，这是 AI 算力架构绝对无法容忍的。

因此，在这个 Agent 系统中，我果断抛弃了分布式中间件，转而利用进程内的 `ArrayBlockingQueue` 结合线程池的 `CallerRunsPolicy` 拒绝策略，实现了一个纯内存的**零拷贝背压引擎**。用极简的单机高并发技巧，就把海量碎块动态凑批喂给本地的 24M 轻量级 Embedding 模型，彻底榨干了机器的极限算力。"

📖 深挖弹药：[11-语义分块与重排版架构](./11-语义分块与重排版架构.md) §动态批处理 + §背压防 OOM

---

### 翻盘路由 ③：被问"JVM / 内存泄漏 / GC" → 引出四层安全沙盒

**触发场景**：面试官问可达性分析、CMS/G1、OOM 排查

**桥接话术**：

"在传统业务中，防内存溢出主要靠 JVM 的垃圾回收调优。但在 AI Agent 时代，内存溢出往往不是因为代码写得不好，而是大模型产生的'**投毒代码**'故意为之——比如引发 fork bomb 或无限内存分配。

传统的 JVM 沙盒或者 SecurityManager 根本防不住这种系统级资源消耗。所以我为 Agent 设计了**四层纵深防御体系**：从应用层的 AST 静态语法树拦截，到 Builtins 白名单，再到底层的 Docker cgroups 物理限额。哪怕大模型生成了极其隐蔽的内存泄漏攻击代码，也会在 256MB 的内存上限处被系统底层的 **OOM Killer 直接斩断**，从而绝对保证了主引擎的存活。"

## 笔记使用姿势 — "以故事开局，以逻辑收尾"

> **核心心法**：你现有的笔记（03/04/05/07/10/11）是优秀的"防御武器库"，但绝对不要主动去背。

### ❌ 错误用法
- 面试官还没问，你就开始："我的系统采用了 ReAct 架构，包含 Think-Act-Observe 三个阶段……"
- 这是"上帝视角的架构宣讲"，面试官听了没有画面感

### ✅ 正确用法
1. **用上面的 SCQA 故事开局** → 面试官被"100 页烂文档"的场景吸引
2. **他一定会追问钩子** → "你刚才提到的 PID 差集法是怎么做的？"
3. **这时候调取笔记逻辑** → 用 03/04/07 笔记里的清晰分点甩给他

**这就是"以具象故事开局，以硬核逻辑收尾"的顶级面试节奏。**

| 阶段 | 你的角色 | 信息来源 |
|------|---------|---------|
| 开局 3 分钟 | 讲故事的人 | 本文的 SCQA 模板 |
| 追问环节 | 答题的人 | 03-11 号笔记的分点逻辑 |
