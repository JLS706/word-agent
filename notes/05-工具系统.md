# 工具系统是怎么设计的？

## 工具的插件化架构是怎样的？

### 具体回答

所有工具继承自 `tools/base.py` 的 `Tool` 抽象基类：

```python
class Tool(ABC):
    name: str = ""            # 工具名称
    description: str = ""     # 功能描述（LLM 看到的）
    parameters: dict = {}     # JSON Schema 格式的参数定义

    @abstractmethod
    def execute(self, **kwargs) -> str:  # 执行逻辑
        ...

    def to_openai_tool(self) -> dict:    # 转为 Function Calling 格式
        ...
```

`ToolRegistry` 是工具注册表，核心功能：
- `register(tool)`: 注册工具（按 name 去重）
- `get(name)`: 按名称查找工具
- `to_openai_tools()`: 批量转为 OpenAI 格式传给 LLM
- `describe()`: 生成文本描述（注入 System Prompt 用）

**设计优点**：新增工具只需继承 Tool、实现 `execute()`、注册即可。不需要修改 Agent 核心代码。

## Agent 能自己创建新工具吗？怎么保证安全？

### 具体回答

可以。`tools/tool_creator.py` 实现了**动态工具创建系统**，配合三层安全审批：

```
Agent 写代码 → Layer 1: AST 安全检查
             → Layer 2: 沙盒试执行（进程隔离）
             → 保存为 .draft 草稿文件
             → Layer 3: 用户手动审批（approve_tool）
             → .draft 重命名为 .py → 注册到 ToolRegistry → 立即可用
```

**三个相关工具**：

| 工具 | 功能 |
|------|------|
| `create_tool` | Agent 写工具代码，经 AST + 沙盒验证后保存为草稿 |
| `approve_tool` | 用户确认代码安全后激活草稿 |
| `list_custom_tools` | 列出所有自定义工具（草稿和已激活） |

**关键安全设计**：
1. Agent 生成的代码**不会直接注册**——必须经过用户审批
2. 草稿以 `.py.draft` 后缀保存，不会被自动加载
3. `approve_tool` 只做文件重命名（`.draft` → `.py`），确保用户已审查过代码
4. 启动时 `load_custom_tools()` 自动扫描 `tools/custom/` 加载已激活的工具

## 工具代码是怎么生成的？

### 具体回答

`_generate_tool_code()` 根据 Agent 提供的参数生成完整的 Python 文件：

```python
# Agent 提供:
tool_name = "kill_word_process"
description = "关闭所有 Word 进程"
parameters = {"type": "object", "properties": {}}
code_body = "import subprocess\nsubprocess.run(['taskkill', '/f', '/im', 'WINWORD.EXE'])\nreturn '已关闭所有 Word 进程'"

# 生成结果:
class CustomTool(Tool):
    name = "kill_word_process"
    description = """关闭所有 Word 进程"""
    parameters = {...}
    def execute(self, **kwargs) -> str:
        import subprocess
        subprocess.run(['taskkill', '/f', '/im', 'WINWORD.EXE'])
        return '已关闭所有 Word 进程'
```

## 沙盒试执行（Layer 2）是怎么测试工具代码的？

### 具体回答

`test_tool_sandboxed()` 的工具模式 Worker：
1. 在隔离进程中执行生成的代码
2. 找到 `CustomTool` 类定义
3. 实例化 `CustomTool()` 
4. 调用 `tool_instance.execute()`（不传参数）
5. 如果抛出 `TypeError: missing required argument` → **结构正常**（说明类定义正确，只是参数缺失）
6. 如果成功返回 → **通过**
7. 其他异常 → **失败**

## 自定义工具怎么持久化的？重启后还在吗？

### 具体回答

**是的**。流程：
1. `create_tool` → 保存为 `tools/custom/xxx.py.draft`
2. `approve_tool` → 重命名为 `tools/custom/xxx.py`
3. 下次启动时，`main.py` 调用 `load_custom_tools(registry)` 扫描 `tools/custom/` 目录
4. 用 `importlib.util.spec_from_file_location()` 动态加载每个 `.py` 文件
5. 查找 `CustomTool` 类 → 实例化 → 注册到 `ToolRegistry`

这是一个**约定优于配置**的设计——只要文件在 `tools/custom/` 目录下且有 `CustomTool` 类，就会被自动加载。

## 动态锚点防护盾是怎么工作的？

> 面试口述中提到的"动态锚点防护盾"，本质是**交叉引用替换时的自我保护机制**，防止工具在执行正文替换时误伤图注标题本身。

### 具体回答

**问题场景**：假设文档中有图注 `图 2.1 系统架构图`，正文中有引用 `如图 2.1 所示`。工具需要把正文的 `图 2.1` 替换成 Word 域代码（REF 书签），但**绝不能动图注标题本身**——否则图注会被破坏。

**难点**：Word 的 `Find.Execute()` 是全篇搜索，它不区分"正文引用"和"图注标题"——两者都包含 `图 2.1` 这个文本。如果不做防护，图注标题会被自己的域代码替换掉，产生循环引用。

**解决方案 — 全局动态锚点防护盾**：

```
替换前：收集所有图注书签 → 记录每个图注所在段落的 Range 范围
替换时：对每个匹配位置，遍历所有图注书签的段落范围
  ├── 匹配位置在某个图注段落内 → 跳过（命中防护盾）
  └── 匹配位置不在任何图注段落内 → 执行替换（安全）
```

**代码核心**（`Word文献自动化精灵.py`）：

```python
# 收集所有图注书签名，用于全局防护盾
all_caption_bkmk_names = list(fig_bookmark_map.values())

# 替换循环中：
for chk_bkmk in all_caption_bkmk_names:
    if doc.Bookmarks.Exists(chk_bkmk):
        chk_rng = doc.Bookmarks(chk_bkmk).Range
        chk_para = chk_rng.Paragraphs(1).Range
        if rng.Start >= chk_para.Start and rng.End <= chk_para.End:
            is_in_any_caption = True  # 命中防护盾 → 跳过
            break
```

**为什么叫"动态"？** 因为书签列表不是硬编码的——每次运行时根据文档实际内容动态收集图注段落。文档增删图注后，防护盾自动更新覆盖范围。

**追问：除了图注防护，还有哪些安全检查？**

替换时还有三层额外防护：
1. **域代码检测**：如果匹配位置已经在一个 REF 域内部 → 跳过（防止域套域）
2. **后缀边界检查**：如果 `2.1` 后面紧跟数字或点（如 `2.13`）→ 跳过（防止短编号吃掉长编号）
3. **前缀安全扫描**：向前逐字符扫描查找 `图`/`Fig.` 前缀，且验证前缀区间不横跨域标记符（防止偏移错位破坏相邻域代码）
