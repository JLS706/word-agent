# 面试 QA — 30 道追问题 + 参考答案

> 按面试官追问的深度排列：**表面问题 → 设计决策 → 压力测试 → 开放性问题**
>
> 每道题标注了**考察意图**，帮你判断面试官到底想听什么。

---

## 一、项目概述（暖场题，必问）

### Q1: 一句话介绍你的项目？
**考察**：表达能力、项目理解深度

**A**: DocMaster Agent 是一个基于 ReAct 架构的学术论文排版智能助手，用户通过自然语言指令驱动 Word 文档自动化处理。核心是手写的 ReAct 推理引擎 + Multi-Agent 三角色协作流水线。

---

### Q2: 为什么做这个项目？不用现有的工具吗？
**考察**：动机、需求洞察

**A**: 学术论文排版有大量重复劳动——参考文献格式统一、交叉引用生成、缩写检查等。Word 自带的功能操作繁琐且容易遗漏。市面上的工具要么是规则式的（不灵活），要么是在线服务（无法处理 Word COM 操作）。我希望做一个能理解自然语言指令、自主决策操作步骤的 Agent。

---

### Q3: 这个项目和直接让 ChatGPT 帮你排版有什么区别？
**考察**：Agent 的核心价值

**A**: ChatGPT 只能给建议，不能直接操作文档。我的 Agent 能**自主调用 Word COM 接口**——比如调用 `format_references` 工具批量修改参考文献的字体和大小写，调用 `create_reference_crossrefs` 生成可跳转的交叉引用域代码。这些操作需要 COM 编程，不是纯文本建议能替代的。

---

## 二、ReAct 架构（核心原理）

### Q4: ReAct 的 Think-Act-Observe 具体怎么实现的？
**考察**：是否真正理解 ReAct 原理

**A**: 在代码层面，Think 和 Act 其实是一次 LLM 调用的两个输出：
- LLM 返回 `tool_calls` → 这就是 Act（决定调用哪个工具、什么参数）
- 我在本地执行工具，把结果以 `role=tool` 消息追加到 history → 这就是 Observe
- LLM 返回纯文本（无 tool_calls）→ 这就是任务完成

循环停止条件：纯文本输出 或 达到 max_steps（默认 10 步）。

---

### Q5: 如果 LLM 陷入死循环怎么办？一直调用同一个工具？
**考察**：边界情况处理

**A**: 三层防护：
1. **max_steps 上限**（默认 10 步）—— 硬限制
2. **Token 水位压缩**——上下文过长时自动截取或摘要旧消息
3. **结构化错误引导**——如果同一工具连续失败 3 次，错误 Observation 会明确告诉 LLM "不要再重复相同调用，请换一种方法"

---

### Q6: 为什么不用 Plan-and-Execute？先制定完整计划不是更可靠吗？
**考察**：架构选型的深度思考

**A**: Plan-and-Execute 的前提是任务可预测，但文档处理有大量不确定性。比如：
- 执行 `format_references` 才会发现某些参考文献格式特殊需要额外处理
- 创建交叉引用时才会发现某些书签已损坏

ReAct 的优势是**每步都能根据观察结果调整策略**。

不过我在 Multi-Agent Pipeline 中也融合了 Plan-and-Execute：Planner 先出计划，Executor 按计划逐步执行——这是两种模式的混合。

---

## 三、Multi-Agent 协作

### Q7: 三个角色共享同一个 LLM，怎么避免角色混乱？
**考察**：Prompt 工程理解

**A**: 通过不同的 System Prompt 切换身份。比如 Planner 的 Prompt 明确写"你只能调用 analyze_document 和 recall_history，不要执行任何修改文档的工具"。Reviewer 的 Prompt 写"你只能调用 read_document 和 inspect_document_format，不能修改文档"。

关键是**工具白名单**——每个角色只能看到允许它使用的工具列表，LLM 的 Function Calling 就不会调用越权的工具。

---

### Q8: 回溯修正的三级策略，为什么不直接跳过失败步骤？
**考察**：设计决策的取舍

**A**: 因为步骤之间可能有依赖。比如 `create_reference_crossrefs`（文献交叉引用）依赖 `format_references`（参考文献格式化）——如果格式化失败就跳过，交叉引用也会失败。

所以优先重试（可能是参数问题），不行再重规划（Planner 可以调整顺序或替换工具），最后才跳过（标记为需人工处理）。

重规划只允许 1 次，防止无限循环浪费 Token。

---

### Q9: Reviewer 真的有用吗？会不会只是浪费 Token？
**考察**：对成本-质量权衡的理解

**A**: Reviewer 的价值在于**发现 Executor 看不到的问题**。比如 Executor 执行完交叉引用后报告"成功"，但 Reviewer 读取文档发现引用指向了错误的段落。

成本控制方面，Reviewer 最多只跑 3 步 ReAct 循环，总消耗约 1000-2000 Token。相比手动检查文档的人力成本，这个 Token 开销是合理的。

---

## 四、记忆系统

### Q10: L1 规则怎么保证 LLM 一定遵守？LLM 不是不可控的吗？
**考察**：LLM 控制的深度理解

**A**: 不存在 100% 保证，但我用了四重保险：
1. **System Prompt 头部注入**——L1 规则在 Prompt 最前面
2. **System Prompt 尾部重复**——首尾夹击，对抗注意力衰减
3. **三明治注入**——在用户消息末尾再次追加关键规则
4. **后校验 `_post_validate_l1()`**——执行后检查结果是否违反 L1，违反则回退

遵从率约 95%，剩下 5% 由后校验兜底。

---

### Q11: L3 到 L2 的晋升机制具体怎么判断？
**考察**：记忆管理细节

**A**: L3 每次被 RAG 召回时 `recall_count += 1`。当 `recall_count >= 5` 时自动晋升到 L2。

晋升时会检查 L2 是否有冲突记忆——用 LLM 做融合判断：
- **REPLACE**：新记忆完全替代旧的
- **MERGE**：LLM 生成融合后的内容
- **KEEP**：保持旧记忆不变

---

### Q12: 向量记忆的召回率淘汰是怎么设计的？
**考察**：工程数学能力

**A**: 每条 L3 记忆有一个效用分数：
```
eviction_score = recall_count × exp(-days_since_last_recall / 30)
```
- `recall_count` 越高 → 越常被使用 → 分数越高
- `days_since_last_recall` 越大 → 越久没被召回 → 指数衰减

当 L3 数量超过上限时，淘汰 `eviction_score` 最低的记忆。这类似于 LRU/LFU 的混合策略。

## 五、Prompt 工程

### Q13: Prompt Cache 的原理是什么？你是怎么利用的？
**考察**：对模型推理优化的理解

**A**: 大模型推理时，Transformer 的注意力计算会生成 KV Cache。如果多次请求的 Token 序列有相同的前缀，厂商可以复用之前计算的 KV Cache，跳过重复计算。

我把 System Prompt 拆成两条消息：
- Message[0]（静态根）：L1 规则 + 工具描述 + 行为准则——跨轮次不变
- Message[1]（动态叶）：技能 + 记忆——每轮变化

Message[0] 的内容始终相同，所以它的 KV Cache 可以被后续请求复用，获得 50%~90% 的输入 Token 降价。

---

### Q14: "动态宪法挂载"是什么意思？为什么不把所有规则都注入？
**考察**：Prompt 优化的细粒度理解

**A**: 如果有 50 条规则全部塞进 Prompt，规则之间会互相干扰。研究表明 Prompt 中的指令越多，单条指令的遵从率越低。

"动态宪法挂载"是根据当前任务的关键词，只选 2-3 条最相关的规则注入。比如用户说"格式化参考文献"，我只注入与参考文献相关的规则，忽略图注、缩写等无关规则。

选择算法用的是关键词映射 + 文本重叠度评分，零 API 调用。

---

## 六、RAG 与语义切块

### Q15: 你的语义切块和 LangChain 的 RecursiveCharacterTextSplitter 有什么区别？
**考察**：对切块策略的理解深度

**A**: RecursiveCharacterTextSplitter 本质还是按字数切——只是递归地尝试不同分隔符（段落→句子→字符）。它不感知语义。

我的语义切块是**基于 Embedding 向量相似度**的：计算相邻句子的余弦相似度，在相似度出现"断崖式下跌"的位置切分。阈值是自适应的（均值 - 1 倍标准差），不需要手动调参。

代价是需要调用 Embedding API，但 Embedding 比 LLM 调用便宜 10 倍以上。

---

### Q16: 为什么要自研 VectorStore 而不用 Pinecone 或 Milvus？
**考察**：技术选型的务实性

**A**: 三个原因：
1. **数据量小**——文档 RAG 通常只有几百条 chunk，余弦相似度暴力搜索在 100ms 内完成
2. **零运维**——不需要维护数据库服务
3. **面试加分**——可以解释余弦相似度的底层原理（NumPy 实现）

如果数据量到百万级，肯定要上 Milvus 或 FAISS。但当前场景下，简单就是最优解。

---

## 七、安全沙盒

### Q17: AST 检查能被绕过吗？你怎么应对？
**考察**：安全意识

**A**: 能。比如 `getattr(__builtins__, '__im' + 'port__')` 可以拼接字符串绕过关键词检查。还有通过 `type.__subclasses__()` 遍历元类链。

这就是为什么我做了四层纵深防御——即使 AST 被绕过：
- Layer 2 的 builtins 白名单会拦截未授权的内置函数
- Layer 3 的进程隔离 + 5 秒超时强杀
- Layer 4 的 Docker 容器隔离

核心设计哲学：**防御深度 > 防御强度**。

---

### Q18: 严格模式和工具模式有什么区别？为什么要区分？
**考察**：安全设计的细粒度

**A**: 
- **严格模式**：给 `execute_python` 用，只允许 `re, math, json` 等纯计算模块，`open()` 只能读，超时 5 秒——因为这是运行 LLM 生成的临时代码
- **工具模式**：给 `create_tool` 的沙盒测试用，额外允许 `os, subprocess, pathlib`，超时 10 秒——因为自定义工具可能需要文件操作

区分的原因：**权限最小化原则**——每个场景只给必需的权限。

## 八、工程实现

### Q19: Token 水位压缩的具体策略？超长对话怎么处理？
**考察**：资源管理

**A**: 两级压缩：
- Tier 1（<6000 Token）：纯规则截取——只保留最近 N 条消息，老消息直接丢弃。零 LLM 成本
- Tier 2（≥6000 Token）：LLM 智能摘要——让 LLM 把历史消息压缩成一段摘要，再替换掉原始消息

优先用 Tier 1，因为大部分场景不需要保留完整历史。只有历史中包含工具结果等关键信息时才触发 Tier 2。

---

### Q20: Checkpoint 为什么用 JSON 而不用 Redis 或 SQLite？
**考察**：技术选型务实性

**A**: 单用户单任务场景——同一时间只有一个 Pipeline 在跑，不存在并发写入问题。JSON 的优势：
1. **零依赖**——不需要安装额外服务
2. **可调试**——直接打开文件就能看到完整状态
3. **可移植**——复制文件就能恢复状态

如果是多用户 SaaS 服务，肯定要用 Redis 做分布式状态管理。

---

### Q21: COM 进程管理有什么坑？你是怎么解决的？
**考察**：真实工程经验

**A**: Word COM 最大的坑是**进程残留**——调用完后 WINWORD.EXE 进程不退出，下次调用会冲突。

我的解决方案：
1. `COMSafeLock`（上下文管理器）：记录创建 COM 时的 PID，退出时精准 kill 该 PID
2. 每个工具调用完后 `close_word` 清理
3. Pipeline 级别的 `_discard_word_changes()` 兜底——Close without save + taskkill 强杀

这是只有实际做过 Windows COM 开发才会遇到的问题。

## 九、Skill 系统

### Q22: Skill 和 Tool 有什么区别？
**考察**：抽象层次理解

**A**: 
- **Tool** 是可执行的代码——有 `execute()` 方法，LLM 通过 Function Calling 调用
- **Skill** 是知识文档——是 `.md` 文件，内容注入到 System Prompt 指导 Agent 的行为

打个比方：Tool 是"锤子"，Skill 是"使用锤子的说明书"。Agent 看了说明书（Skill），知道该用哪把锤子（Tool）、按什么顺序用。

---

### Q23: Skill 的 config 合并策略是什么？如果两个 Skill 冲突怎么办？
**考察**：配置管理设计

**A**: 深度合并，高优先级覆盖低优先级。比如：
- 通用 Skill：`{正文: {font_cn: "宋体", line_spacing: 1.5}}`
- B 校 Skill（优先级更高）：`{正文: {font_cn: "仿宋"}}`
- 合并结果：`{正文: {font_cn: "仿宋", line_spacing: 1.5}}`

B 校 Skill 只覆盖了字体，行距自动使用通用规范兜底。如果某字段设为 `null`，表示"不检查这个字段"。

## 十、开放性问题（压力测试）

### Q24: 如果要把这个系统部署到 Linux 上，最大的障碍是什么？
**考察**：系统认知、迁移能力

**A**: 最大的障碍是 **Word COM 接口**——`pywin32` 只能在 Windows 上运行。迁移方案有两条路：
1. **python-docx 替代**——python-docx 是纯 Python 的 docx 操作库，跨平台。但它不支持域代码、VBA 宏、MathType 等高级功能
2. **LibreOffice UNO 接口**——Linux 上可以用 LibreOffice 的 UNO API 代替 Word COM，功能较完整但接口风格完全不同

我的架构中工具层和核心层是分离的，理论上只需要替换 `tools/` 目录下涉及 COM 的文件，核心引擎不需要改动。

---

### Q25: 如果用户量从 1 人变成 100 人，架构要怎么改？
**考察**：扩展性思维

**A**: 主要改三个地方：
1. **任务队列**——引入 Celery/RQ，API 收到请求后放入队列，Worker 异步执行
2. **状态管理**——Checkpoint 从 JSON 文件迁移到 Redis
3. **资源隔离**——每个用户的 Word 进程需要隔离，可以用 Docker 容器池

Agent 核心引擎不需要改，因为它本身是无状态的——state 都在 Memory 和 Checkpoint 里。

---

### Q26: 和 LangChain Agent 相比，你的手写 Agent 有什么劣势？
**考察**：自我认知、诚实度

**A**: 坦诚说有几个劣势：
1. **生态**——LangChain 有现成的 Document Loader、Vector Store、Memory 集成，我的都是自己写的
2. **Streaming**——LangChain 原生支持 token-level streaming，我还没实现
3. **社区**——遇到问题有大量参考资料，我的就靠自己

但手写的优势是**完全可控且面试友好**——每一行代码都是我自己的设计决策，面试官追问任何细节我都能答。

---

### Q27: 你觉得这个项目最大的技术难点是什么？
**考察**：深度自省

**A**: **让 LLM 可靠地遵守规则**。LLM 的输出是概率性的，不像传统程序可以写 `if-else` 保证行为。

比如 L1 规则"每次调用 Word 工具后必须关闭进程"——LLM 有时候会忘记。我为此设计了四重保险（头部注入、尾部重复、三明治注入、后校验），遵从率才达到 ~95%。

这让我深刻体会到：**工程化 LLM 应用的核心挑战不是调 API，而是控制不确定性**。

---

### Q28: 如果重新做这个项目，你会做哪些不同的设计？
**考察**：反思能力

**A**: 三个改进：
1. **异步化**——当前是同步阻塞的，如果引入 `asyncio` 可以实现工具并行调用
2. **可观测性**——接入 Langfuse 或 LangSmith，追踪每步的 Token 消耗和延迟
3. **评测体系**——建立自动化基准测试，量化每次改动对成功率的影响

---

### Q29: 你怎么测试 Agent 的质量？有没有评测指标？
**考察**：质量意识

**A**: 当前是手动验证为主——Pipeline 跑完后 Reviewer 出报告（S/A/B/C/D 评分），我人工确认。

理想的评测体系应该包括：
1. **工具调用准确率**——给定指令，看 Agent 是否调用了正确的工具
2. **参数正确率**——工具参数是否合理
3. **端到端成功率**——给一篇待排版论文，Pipeline 一跑到底的成功率
4. **L1 规则遵从率**——L1 规则是否被违反

目前没有自动化评测是因为文档处理的"正确性"很难机器判断——需要人工 review。

---

### Q30: 这个项目用到了哪些设计模式？
**考察**：软件工程基础

**A**:
1. **模板方法模式**——`Tool` 抽象基类定义 `execute()` 接口，子类实现具体逻辑
2. **注册表模式**——`ToolRegistry` 统一管理所有工具的注册和查找
3. **策略模式**——错误分类后选择不同的恢复策略（重试/引导/放弃）
4. **状态机模式**——`WorkflowPhase` 枚举驱动 Pipeline 的阶段跳转
5. **责任链模式**——安全沙盒的 4 层防护，每层只关注自己的职责
6. **观察者模式**——子任务反思 Hook，每步完成后自动触发经验提炼
