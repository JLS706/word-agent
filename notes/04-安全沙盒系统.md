# 安全沙盒是怎么设计的？

## 为什么需要沙盒？Agent 执行代码有什么安全风险？

### 具体回答

Agent 拥有 `execute_python` 工具，可以让 LLM 生成代码并执行。如果不加限制：
- LLM 可能生成 `os.system("rm -rf /")` 删除系统文件
- 可能通过 `requests` 发送网络请求泄露数据
- 可能陷入死循环耗尽 CPU 资源
- 可能通过 `pickle`/`marshal` 实现反序列化攻击

所以需要一个**纵深防御**架构——不依赖单一防线，任何一层被绕过还有下一层兜底。

## 四层防护架构具体是怎样的？

### 具体回答

```
代码进入 → Layer 1: AST 静态分析（编译前拦截）
         → Layer 2: builtins 白名单（运行时约束）
         → Layer 3: 进程隔离 + 超时强杀（资源兜底）
         → Layer 4: Docker 容器隔离（OS 级沙盒）
```

**Layer 1：AST 静态分析** (`SafetyChecker`)
- 用 `ast.parse()` 把代码解析成抽象语法树
- 遍历 AST 节点，检查 `Import`、`ImportFrom`、`Call`、`Attribute`
- 白名单机制：只允许 `re, math, json, collections, datetime` 等安全模块
- 黑名单机制：禁止 `socket, http, requests, ctypes, pickle` 等
- 禁止危险调用：`exec()`, `eval()`, `__import__()`, `compile()`
- 禁止危险属性访问：`os.system()`, `.remove()`, `.unlink()` 等

**Layer 2：builtins 白名单** (`_make_safe_builtins`)
- 把 `__builtins__` 替换为只包含安全内置函数的字典
- 移除 `exec`, `eval`, `__import__` 的直接访问
- 严格模式下 `open()` 被替换为只允许 `mode='r'` 的安全版本
- 提供受限的 `__import__`：只能导入白名单内的模块

**Layer 3：进程隔离 + 超时强杀** (`_run_in_process`)
- 用 `multiprocessing.Process(daemon=True)` 创建子进程
- 设置超时（严格模式 5 秒，工具模式 10 秒）
- 两级终止策略：`terminate()` → 等 2 秒 → `kill()` → 等 1 秒
- 即使代码绕过了 Layer 1 和 2，也无法长时间占用资源

**Layer 4：Docker 容器隔离**
- 通过 `SANDBOX_URL` 环境变量配置 Docker 沙盒微服务
- 代码通过 HTTP 发送到独立容器执行（而非主进程）
- 容器资源限制：1 CPU / 256MB 内存
- 如果 Docker 不可用，自动回退到本地 Layer 1-3

## 严格模式和工具模式有什么区别？

### 具体回答

| 维度 | 严格模式 (`strict`) | 工具模式 (`tool`) |
|------|--------------------|--------------------|
| 用途 | `execute_python` 工具 | `create_tool` 测试 |
| 白名单 | `re, math, json, collections` 等纯计算模块 | 额外允许 `os, subprocess, pathlib, shutil` |
| 文件操作 | `open()` 只允许读 (`mode='r'`) | 允许写入 |
| 超时 | 5 秒 | 10 秒 |
| 设计思路 | 只读分析，零副作用 | 需要系统操作，但仍在沙盒内 |

## AST 检查能被绕过吗？为什么必须有 Layer 2？

### 具体回答

AST 之所以防不住，是因为它只能看到**语法结构**，看不到**运行时状态**。在 Python 中，大模型哪怕不用 `import os`，照样可以通过两种经典的动态特性直接穿透 AST：

**绕过方式一：动态属性获取（getattr 字符串拼接）**

AST 可以拦截明文的 `eval()` 或 `__import__()` 调用。但如果大模型写出这样的代码：

```python
func = getattr(__builtins__, '__im' + 'port__')
os_mod = func('o' + 's')
os_mod.system('rm -rf /')
```

在 AST 看来，这只是一次普通的 `getattr` 函数调用和字符串加法，完全合法！AST 根本不知道它在运行时拼出来的是恶意的系统导入。

**绕过方式二：Python 元类链与反射（沙箱越狱）**

这是更高级的攻击。在 Python 里，随便找一个基础对象（比如空元组 `()`），顺藤摸瓜就能爬到顶层的 `object` 基类，然后遍历所有子类，直接把躲在暗处的危险模块给揪出来：

```python
().__class__.__bases__[0].__subclasses__()
```

只要大模型遍历这个子类列表，找到类似 `subprocess.Popen` 或 `os._wrap_close` 这样的类，它就能在**完全不使用 import 关键字**的情况下，直接拉起系统命令行执行危险操作。AST 对这种连续的属性访问（`.` 操作符）根本无能为力——如果全拦截了，正常的面向对象代码也就废了。

**所以，这就是 Layer 2（Builtins 白名单）存在的必要性**

既然防不住大模型用动态手段去"找门"，那就直接在运行时的字典里**把门焊死**。我接管了执行上下文的 `restricted_globals`，把 `__builtins__` 里的 `__import__`、`eval` 全部强行剔除，甚至把 `open` 函数替换成了只能读（`mode='r'`）的残割版。这样一来，哪怕大模型用尽一切反射手段，当它在运行时试图调用时，底层的 Python 解释器只会冷冷地抛给它一个 `NameError` 或 `PermissionError`。

**纵深防御的整体逻辑**：
- **Layer 1（AST）** 拦截 90% 的明文危险代码
- **Layer 2（Builtins 白名单）** 在运行时焊死剩余 10% 的动态绕过路径
- **Layer 3（进程隔离）** 保证即使代码成功执行，也无法影响主进程
- **Layer 4（Docker 隔离）** 保证即使逃逸子进程，也无法影响宿主机

这是**防御深度 > 防御强度**的设计哲学。

## Docker 沙盒微服务的架构是怎样的？

### 具体回答

```
┌─── Docker 内部网络 ───────────────────┐
│                                       │
│  agent (8000 对外)                    │
│    ├── FastAPI + ReAct Agent          │
│    └── 调用 sandbox 执行代码           │
│           ↕ HTTP 内部通信              │
│  sandbox (8001 仅内部)                │
│    ├── 独立沙盒微服务                  │
│    └── 资源限制: 1 CPU / 256MB        │
│                                       │
└───────────────────────────────────────┘
```

- Agent 容器通过 HTTP POST 把代码发给 Sandbox 容器
- Sandbox 容器有两个端点：`/execute`（严格模式）和 `/test-tool`（工具模式）
- Sandbox 端口 8001 **不对外暴露**，只在 Docker 内部网络可访问
- 如果 Sandbox 容器不可用，Agent 自动回退到本地多进程沙盒
