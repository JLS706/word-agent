# DocMaster Agent - 架构设计问答集

## Q1: 针对大模型长上下文注意力涣散的问题，为什么一定要采用"长短期记忆"的方式来解决，而不是通过优化文档结构设计或其他手段来解决？

**A:** 优化文档结构（RAG 和切块）只能解决"文档内容"的上下文超载；而分层记忆和反思，解决的是 Agent 在漫长且复杂的试错生命周期中，"控制流与行为准则"的遗忘和进化问题。一个是处理静态的数据，一个是管理动态的灵魂。

具体来说：
- **文档结构优化**：解决的是输入文档的信息密度和检索效率问题
- **长短期记忆系统**：解决的是 Agent 在多轮交互中保持行为一致性、学习用户偏好、避免重复错误的问题

## Q2: 针对"忘记关闭Word僵尸进程"的问题，处理文档是一个标准化流程，为什么不能直接沉淀为一个Skill（技能/标准化流程）每次自动加载执行，而要作为记忆规则注入？

**A:** 这个问题直击本质，也是我在架构演进中反思最深的一点。

### 初期的错误尝试
一开始，我确实把类似'执行完必须关闭 Word 进程'这样的指令写在了 L1 核心记忆里，试图用 Prompt 去约束模型。但后来在压测时我发现，**大模型本质上是一个概率引擎，用概率去对抗确定性的物理资源回收，是一定会出 bug 的**。

### 架构重构：物理的归物理，认知的归认知
所以我立刻做了重构：

#### 1. 物理层兜底（确定性保障）
对于'防进程残留'这种绝对刚性需求，我直接下沉到了代码层：
- 自研了 `COMSafeLock` 上下文管理器，提供进程隔离 + PID 精准狙击
- Agent 每次 run 结束时加了一个硬编码的 `_post_validate_l1()`，一旦检测到用了 Word 工具但没关进程，代码会自动补上一刀，绝对不信任大模型

#### 2. 认知层对齐（让大模型知道物理法则）
既然底层已经兜底了，为什么我还坚决保留了 L1/L2/L3 的三层记忆系统？

**原因一：维持 Agent 认知与物理环境的对齐（纵深防御）**
如果只在底层偷偷杀进程，大模型的状态机就和物理世界脱节了。L1 记忆里的规则，是为了让大模型'知道'这个物理法则的存在，从而在它自己输出的 tool_calls 计划里就能形成逻辑闭环。

**原因二：处理无法代码硬编码的柔性对齐问题**
三层记忆的核心阵地，是解决那些'无法被代码硬编码'的柔性对齐问题。比如：
- 每个学者对排版的容忍度、习惯、报错后的策略偏好是完全不同的
- 系统里有 `save_learned_rule` 工具，Agent 可以在一次次报错和与用户的交互中，自己总结出'这位用户的图注喜欢用半角括号'，然后写入 L2 向量记忆或 L1 铁律

### 设计哲学
- **物理层**：处理确定性、可硬编码的问题（进程管理、文件备份、异常回滚）
- **认知层**：处理不确定性、需要学习和适应的问题（用户偏好、领域知识、策略选择）

这种分层设计确保了系统的**可靠性**（物理层兜底）和**智能性**（认知层进化）的平衡。
