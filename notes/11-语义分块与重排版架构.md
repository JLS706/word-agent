# 生产级多智能体系统：混沌长文本的语义分块与重排版架构演进

## 一、 业务背景与初始痛点

**场景描述：**
用户上传一份长度极长（例如 20 万字）、缺乏逻辑标题、排版极其混乱的"混沌纯文本"或代码混杂文本。系统需要将其自动结构化，并输出带有清晰层级（如大标题、小标题、核心摘要）的规范化 Word 文档。

**初始方案（痛点暴露）：**
1.  **强行按标点切分：** 简单的按句号切分会导致极度碎化的"语义断层"，上下文联系被破坏。
2.  **直接丢给大模型处理：** 将海量碎块直接送入云端重型 LLM 计算相似度，会导致极高的网络 I/O 延迟、触发 API 的 QPS 限制，甚至导致单次推理成本失控。

---

## 二、 核心破局思路：大小模型协同

该架构的本质思想是：**"脏活累活本地做，高阶总结云端做。"**

本地小模型（24M 参数）负责把几千个碎块缩减成几十个宏观段落（计算密集，但不需要智能）；云端大模型只处理这几十个段落的标题生成和摘要提炼（需要智能，但量已经很小，成本可控）。

### 完整流程总览

```
Stage 1-2: 本地完成（切块 + 算向量 + 找语义边界）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌───────────────────┐   ┌────────────────┐   ┌─────────────────────────┐

│ ↓                 │   │   [碎块5]       │←取│  queue.take()           │
```

## 三、 生产级工程优化（核心考点）

在普通服务器（8 核 CPU、16G 内存，无 GPU）上处理海量文本，需要三个关键机制配合。

### 1. 零拷贝集成 (Zero-Copy Integration)

严禁通过 HTTP 跨进程调用 Python 封装的模型。必须使用 **DJL (Deep Java Library)** 或 **ONNX Runtime for Java**，Java 对象直接与 C++ 推理引擎共享物理内存，彻底消除序列化与网络传输损耗。

*比喻：不再是写信并通过邮局把数据发给对面的同事（序列化+网络传输），而是两人直接共用一块白板（共享内存）。*

### 2. 动态批处理 (Dynamic Batching)

Embedding 模型底层是矩阵乘法——计算 1 个向量和计算 10 个向量的耗时几乎一样（SIMD 并行特性）。所以工人线程不会取 1 个碎块就推理，而是**按 Token 预算动态凑批**：

```
工人线程从队列取碎块:
  取碎块1 (200 tokens) → 累计 200，还没满 512
  取碎块2 (150 tokens) → 累计 350，还没满
  取碎块3 (180 tokens) → 累计 530，接近上限，够了！
  → 打包 [碎块1+2+3] 送进小模型 → 一次矩阵推理 → 拿回3个向量
```

每批大小**不固定**：短碎块凑多个，长碎块凑少个，目标是填满模型的上下文窗口（如 512 Tokens）。这就是"动态"的含义。

*比喻："拼车"逻辑，等座位快坐满了再发车，榨干每次运算的算力。*

### 3. 背压防 OOM 机制 (Backpressure)

**核心矛盾：** 切块速度（几毫秒/个）远快于模型推理速度（几十毫秒/批），队列必然会被填满。

**解决方案：** 有界阻塞队列 + `CallerRunsPolicy` 拒绝策略。

```
正常状态:
  主线程切碎块 → 放入队列 → 工人线程取走 → 推理（队列缓缓填满）

队列满了（背压触发）:
  主线程切了一个碎块 → 想放入队列 → 队列满了！

  CallerRunsPolicy: "放不进去？那你主线程自己去跑推理！"

  主线程被迫去干推理（耗时长）
    → 没人切新碎块了 → 生产停滞
    → 工人线程消费了一些碎块 → 队列腾出空位
    → 主线程推理完 → 回来继续切碎块

  = 物理层面自动限流，无需限流算法
```

*比喻："包工头效应"。候车室（队列）满了，不是拒绝乘客，而是强行命令包工头（主线程）亲自去开车。包工头去开车了，自然就停止了切分新文件。*

### 4. 三者的关系与线程模型（易混淆）

> **常见疑问：文本块不是已经切好了么，为什么还需要缓冲队列和批组装？**

它们在不同层面解决不同问题：

| | 语义分块 | 缓冲队列 | 动态批处理 |
|---|---|---|---|
| **本质** | 主线程的工作逻辑 | 共享数据结构（不是线程！） | 工人线程的工作逻辑 |
| **目的** | 按语义边界切刀 | 限制排队数量，防 OOM | 凑批提升推理效率 |
| **去掉会怎样** | 切割无语义 | 碎块堆积 → OOM | 逐个推理 → 慢 10 倍 |

**一句话总结：切块决定语义质量，批处理决定计算速度，缓冲队列决定系统能不能活着跑完。**

## 四、 核心算法：双重约束的贪心聚合 (Greedy Aggregation)

在本地小模型算完所有 Embedding 向量后，如何将碎块拼装成宏观段落？采用带有双重约束的 $O(N)$ 贪心算法：

维护一个"当前拼装栈（Accumulator）"。对于涌入的新碎块：

1.  **约束一：Token 容量探针。** 若 `当前栈 Token + 新碎块 Token > 大模型最大限制`，则**强制弹出**当前栈，新碎块作为下一个段落的起点。
2.  **约束二：语义连续性探针。** 若未超容量，计算相似度。若相似度高，则**压栈合并**；若发生"语义滑坡"（相似度低于 $\mu + c \cdot \sigma$），则即便未满也**强制弹出**当前栈。

> **补充优化：规避云端大模型 Lost in the Middle**
> 即使分块完毕，拼接后的长文本直接丢给 LLM 也容易导致中部内容丢失。
> 方案：在交给云端前，注入强结构的 XML 锚点（如 `<section id="001">...</section>`），并在 Prompt 中下达按 ID 处理的铁律。最后采用 Map-Reduce 树状归并策略，先局部提炼，再全局总结。
