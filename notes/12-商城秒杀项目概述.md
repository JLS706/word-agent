# 商城秒杀项目概述

> 这是你的**第二个面试项目**。Agent 项目展示 AI 工程能力，秒杀项目展示**后端工程能力**。
> 两个项目互补，面试官问哪个都有东西可聊。

---

## 一句话介绍

基于 Spring Boot 3.x 的电商秒杀系统，在 newbee-mall 开源项目基础上，**独立设计并实现了完整的高并发秒杀模块**——Redis Lua 原子预扣 + RabbitMQ 异步下单 + MySQL 行锁兜底的三层防超卖架构。

## 你改了什么？（和原版 newbee-mall 的差异）

> 这是面试官最看重的——不是"你用了什么项目"，而是"你在里面做了什么"。

| 模块 | 原版 | 你的改造 | 为啥改 |
|------|------|---------|--------|
| 防超卖 | 无（或简单的 DB 扣减） | Redis Lua 原子预扣 + DB 行锁兜底 | 纯 DB 扣减扛不住并发 |
| 下单 | 同步创建订单 | RabbitMQ 异步削峰 | 同步写 DB = 用户等半天 + DB 被打爆 |
| 订单号 | 时间戳 + 4 位随机数 | 手写雪花算法 | 高并发下随机数会重复 |
| Redis-DB 一致 | 无 | Consumer 失败时回补 Redis | 不回补 = 库存凭空消失 |
| 事务管理 | `@Transactional` 直接加 | 拆到独立方法（避免同类调用失效） | 踩了 Spring AOP 代理的坑 |
| 库存预热 | 无 | `CommandLineRunner` 启动即加载 | 秒杀开始时 Redis 必须有库存 |

## 面试口述模板 — 极端场景版（面试官出题时用）

> 面试官问："假设 10 万人同时抢 100 台限量手机，你怎么保证不超卖？"
>
> **核心原则**：每一句话都有代码兜底，不说没做过的东西。

---

### 完整话术（~2 分钟）

"面试官您好。10 万并发抢 100 台手机，本质上是个**漏斗型流量拦截**的问题——我的目标是在尽可能前端的位置就把无效请求拦掉，最终只让真正抢到的 100 个请求去碰数据库。

我的系统有**两道核心防线 + 一层异步缓冲**：

**第一道防线：Redis Lua 脚本 — 在内存里决出胜负。**

活动开始前，我通过 `CommandLineRunner` 在项目启动时就把 100 台手机的库存预热到 Redis。当 10 万请求打进来时，我不碰 MySQL，直接向 Redis 发送一段 Lua 脚本。这段脚本在 Redis 单线程里**原子地**做三件事：
1. **查用户有没有买过**——一个 EXISTS 命令，防止同一个人重复抢；
2. **查库存够不够**——GET 库存 key，小于等于 0 直接返回'卖光了'；
3. **扣库存 + 标记用户**——DECR 库存，SETEX 标记用户已购买。

因为 Redis 单线程执行 Lua，不需要任何分布式锁，**物理上就保证了原子性**。当第 101 个请求执行时，脚本看到库存 ≤ 0，直接返回 -1，后端立刻响应'已售罄'。这一层瞬间拦掉了 99.9% 的请求。

**第二道防线：RabbitMQ 异步削峰。**

经过 Lua 脚本的无情筛选，只有 100 个抢到的请求拿到了'通行证'。这时候我才把 userId + seckillId 丢进 RabbitMQ 队列——MQ 处理的压力从 10 万瞬间降到了 100 条。Consumer 在后台逐条消费，创建订单、写订单项、扣 MySQL 真实库存。MySQL 的 SQL 还带了一个兜底条件 `WHERE seckill_num > 0`，即使前面出了任何幺蛾子，数据库行锁也能保证绝不超卖。

**另外有一个特别容易被忽略的工程细节**——如果 Consumer 在创建订单时 MySQL 写失败了怎么办？Redis 库存已经扣了，不回补的话库存就凭空消失了。所以我在 Consumer 的 catch 里做了两步回补：**Redis 库存 +1 恢复，同时清除用户购买标记**，让这个用户和这件库存都有机会重新参与。

总结来说：**用 Redis Lua 在内存里决胜负，用 MQ 做削峰稳落库，用 DB 行锁做最后保险丝，用回补机制保 Redis-MySQL 一致性**。这四层组合确保不会卖出第 101 台手机。"

## 面试口述模板（3 分钟版）— SCQA 泥坑叙事

> 面试官问："介绍一下你的秒杀项目？"

### 第一段：定调（~30 秒）

"我的第二个项目是一个电商秒杀系统，基于 Spring Boot 3 和 newbee-mall 开源项目。原版只有基础的商城功能，**没有秒杀模块**。我独立设计了整个秒杀链路——从前端点击到最终生成订单，核心要解决的问题就一个：**怎么在高并发下保证不超卖**。"

### 第二段：讲链路（~80 秒）

"防超卖这件事，我做了**三层防护**：

第一层在 **Redis**。用户请求进来，我不直接碰数据库，而是先执行一段 **Lua 脚本**。这段脚本在 Redis 里一次原子操作做三件事——查用户有没有买过、查库存够不够、扣库存标记用户。整个过程 Redis 单线程执行，不需要分布式锁，天然不会出现并发冲突。

第二层是 **MQ 异步削峰**。Redis 预扣成功后，我不是直接去写 MySQL——如果 1000 个人同时过了 Redis 这关，MySQL 直接被打爆。所以我把请求丢进 RabbitMQ，Consumer 在后台慢慢消费、逐条创建订单。用户那边秒级就拿到'排队中'的响应了。

第三层是 **MySQL 兜底**。Consumer 扣库存时 SQL 带着 `WHERE seckill_num > 0`，即使 Redis 和 MQ 出了什么幺蛾子，MySQL 的行锁也能保证不超卖。

另外有个特别容易被忽略的坑：**如果 MySQL 写失败了，Redis 里的库存已经扣了，不回补的话库存就凭空消失了**。所以我在 Consumer 的 catch 里做了 Redis 库存回补 + 用户标记清除。"

### 第三段：亮工程细节（~50 秒）

"除了秒杀防护，还有几个我比较得意的工程设计：

订单号用的是我**手写的雪花算法**——64 位 long，时间戳 + 机器 ID + 序列号位运算拼接，每毫秒 4096 个 ID 不重复，替掉了原版的时间戳+随机数方案。

库存预热用 `CommandLineRunner`，项目启动时自动把数据库里的秒杀商品库存加载到 Redis，不需要手动操作。

还有一个踩坑的地方——`@Transactional` 同类调用失效。我一开始把事务注解直接加在 Consumer 的消费方法上，后来发现事务根本没生效，排查了半天才明白是 Spring AOP 代理的问题，最终把数据库操作抽到了独立方法里。"

## 核心设计决策速查表

> 面试官问"为什么用 X 不用 Y"时的弹药库

| 决策 | 选了什么 | 没选什么 | 一句话理由 |
|------|---------|---------|-----------|
| 防并发 | Lua 脚本原子操作 | 分布式锁（Redisson） | Lua 天然原子，性能更高，无锁等待开销 |
| 下单方式 | MQ 异步 | 同步写 DB | 同步 = 用户等 + DB 被打爆 |
| 订单号 | 雪花算法 | UUID / 数据库自增 | UUID 无序影响索引性能，自增暴露信息 |
| 数据库 | MySQL 行锁兜底 | 不做 DB 校验 | Redis 不是 100% 可靠，必须有最后防线 |
| 持久化格式 | RDB + AOF 混合 | 纯 RDB 或纯 AOF | RDB 快但丢数据多，AOF 全但恢复慢，混合取长补短 |
