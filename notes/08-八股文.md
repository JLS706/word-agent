# AI Agent 核心知识点（八股文）

> 每个知识点都标注了 **DocMaster 中的对应实现**，面试时可以直接说"我在项目中是这样做的"。

---

## 1. Agent 与 Workflow 的区别

**概念区分**：
- **Workflow**（工作流）：预定义的固定流程，节点和边在代码中硬编码，运行时不变
- **Agent**（智能体）：LLM 自主决策调用哪些工具、以什么顺序，运行路径是动态的

**关键判断标准**：如果把 LLM 换成 `if-else`，逻辑还能跑吗？
- 能 → Workflow（伪 Agent）
- 不能 → 真 Agent（决策依赖 LLM 推理）

**DocMaster 对应**：
- `core/agent.py` 的 ReAct 循环 → **真 Agent**（LLM 自主选择工具和参数）
- `core/multi_agent.py` 的 Pipeline → **Workflow**（Planner→Executor→Reviewer 顺序固定）
- 两者结合：**宏观 Workflow + 微观 Agent**——Pipeline 是固定的，但每个阶段内部的工具调用是 LLM 自主决策

## 2. ReAct vs CoT vs Plan-and-Execute

| 架构 | 核心思想 | 适用场景 | 局限 |
|------|----------|----------|------|
| **CoT** (Chain-of-Thought) | 让 LLM 分步推理 | 数学、逻辑题 | 不与外部环境交互 |
| **Plan-and-Execute** | 先出完整计划，再一次性执行 | 确定性任务 | 执行中无法动态调整 |
| **ReAct** (Reason + Act) | Think→Act→Observe 循环 | 需要工具交互、结果不确定 | 可能陷入循环 |

**DocMaster 选 ReAct 的原因**：
1. 文档处理结果不确定——格式化后可能有遗漏，需要观察结果再调整
2. 天然兼容 Function Calling——`tool_calls` = Act，纯文本 = Finish
3. 代码：`agent.py` 的 `run()` 方法，约 100 行实现完整 ReAct 循环

## 3. Multi-Agent 编排模式

### 常见编排模式

| 模式 | 说明 | 示例 |
|------|------|------|
| **Pipeline（串行）** | A → B → C 顺序执行 | DocMaster: Planner→Executor→Reviewer |
| **Parallel（并行）** | A, B, C 同时执行 | 航班/酒店/活动并行搜索 |
| **DAG（有向无环图）** | 依赖关系决定执行顺序 | LangGraph |
| **Loop（循环）** | 条件不满足时回到上游 | DocMaster: 回溯修正 |
| **Hierarchical（层级）** | 上级 Agent 分派任务给下级 | CrewAI |

**DocMaster 的编排**：Pipeline + Loop（回溯修正）
- Pipeline 保证流程有序
- 回溯修正提供容错：失败 → 重试 → 重规划 → 跳过

## 4. Function Calling 原理

**核心流程**：
```
开发者定义 tools（JSON Schema） → 传给 LLM
LLM 推理后返回 tool_calls（工具名 + 参数）
开发者在本地执行工具 → 结果以 role=tool 消息回传 LLM
LLM 根据结果继续推理或输出最终答案
```

**关键细节**：
- LLM 不执行工具，只"建议"调用——执行在开发者侧
- 参数由 JSON Schema 约束（类型、必填字段、枚举值）
- 多工具可以并行调用（`parallel_tool_calls`）

**DocMaster 对应**：
- `tools/base.py` 的 `Tool.to_openai_tool()` → 转为 JSON Schema
- `core/agent.py` 的 `_execute_tool()` → 本地执行
- `core/schema.py` 的 `ToolCall` → 结构化表示

## 5. Prompt Engineering 进阶技巧

### 5.1 Prompt Cache（前缀缓存）
- **原理**：大模型 API 缓存 Token 序列的最长公共前缀的 KV Cache
- **要求**：System Prompt 内容跨轮次不变
- **DocMaster 实现**：静态根（L1+工具+行为准则）+ 动态叶（技能+记忆）分离，`prompt.py` 的 `build_static_system_prompt()` vs `build_dynamic_context()`

### 5.2 三明治注入（Sandwich Injection）
- **问题**：LLM 对长 Prompt 的中间位置注意力最弱（Lost in the Middle）
- **解法**：将关键规则放在 Prompt 的头部和尾部（首尾夹击）
- **DocMaster 实现**：L1 规则头部注入 + `build_l1_user_suffix()` 尾部注入

### 5.3 动态宪法挂载（Dynamic Constitution）
- **问题**：50 条规则全塞进 Prompt 会互相干扰
- **解法**：按任务关键词只注入 2-3 条最相关的规则
- **DocMaster 实现**：`prompt.py` 的 `select_relevant_rules()`

## 6. RAG（检索增强生成）

### 核心流程
```
文档 → 切块 → Embedding → 向量存储
查询 → Embedding → 余弦相似度检索 → Top-K → 注入 Prompt → LLM 生成
```

### Naive RAG vs Advanced RAG

| 维度 | Naive RAG | DocMaster 的做法 |
|------|-----------|-----------------|
| 切块 | 固定 500 字一刀 | **语义切块**（Embedding 相似度断崖检测） |
| 检索 | 单一向量检索 | 双层匹配（关键词 + Embedding） |
| 数据源 | 只做文档 RAG | 文档 RAG + **记忆 RAG**（对话历史也向量化） |

**DocMaster 对应**：
- `core/semantic_chunker.py` → 语义切块（自适应阈值 μ-1σ）
- `core/embeddings.py` → 自研 VectorStore（纯 NumPy）
- `tools/rag.py` → IndexDocumentTool + SearchDocumentTool
- `core/memory.py` → 记忆 RAG（对话 Q+A 向量化后检索）

## 7. 记忆系统设计

### 为什么需要分层？
- **短期记忆**：当前对话的上下文（LLM 窗口自带）
- **长期记忆**：跨会话的经验积累（需要显式持久化）
- **核心记忆**：不可违反的铁律（需要强制注入）

### DocMaster 三级记忆

| 层级 | 特征 | 过期策略 |
|------|------|----------|
| L1 核心规则 | 铁律，永不过期 | 用户手动管理 |
| L2 长期反思 | 向量化经验 | FIFO 50 条上限 + 冲突融合 |
| L3 短期对话 | 问答摘要 | 30 天 TTL + 召回率淘汰 |

### 记忆晋升机制
- L3 某条记忆的 `recall_count ≥ 5` → 自动晋升到 L2
- L2 与已有记忆冲突 → LLM 融合判断（REPLACE / MERGE / KEEP）

## 8. 安全沙盒设计（纵深防御）

### 为什么单层防护不够？
- AST 检查可被字符串拼接绕过
- builtins 白名单可被元类链遍历绕过
- 进程隔离可被 fork bomb 攻击

### 纵深防御 = 每一层都假设上一层被绕过

```
Layer 1: AST 静态分析     → 编译前拦截（白/黑名单）
Layer 2: builtins 白名单   → 运行时约束（移除危险内置函数）
Layer 3: 进程隔离+超时强杀  → 资源兜底（5秒超时）
Layer 4: Docker 容器隔离   → OS 级隔离（1CPU/256MB）
```

**DocMaster 对应**：`core/sandbox.py`（Layer 1-3）+ `sandbox/` 目录（Layer 4 Docker）

## 9. Token 管理与成本优化

### 上下文溢出问题
LLM 有 Token 上限（如 128K），长对话会导致溢出。

### DocMaster 的两级压缩

| 水位 | 策略 | 成本 |
|------|------|------|
| < 4000 Token | 不压缩 | 零 |
| 4000~6000 | Tier 1: 截取最旧消息 | 零 |
| ≥ 6000 | Tier 2: LLM 智能摘要历史 | ~600 Token |

**设计原则**：优先用零成本的规则方法，只有规则方法不够时才调 LLM。

## 10. Checkpoint（断点续传）

**问题**：Multi-Agent Pipeline 可能跑 10 分钟，中途 Word 崩溃了怎么办？

**解法**：类似游戏存档——每完成一个关键节点自动序列化状态到 JSON 文件

**DocMaster 实现**：
- `WorkflowPhase` 枚举：9 个阶段状态
- `WorkflowState`：完整快照（阶段、计划、已完成步骤、重试计数）
- `Checkpointer`：save / load / clear / list

**为什么用 JSON 而不是 Redis？** 单用户单任务场景，JSON 简单可靠、便于调试。

## 11. 工具自治与自我进化

**概念**：Agent 不仅使用工具，还能**创造工具**。

**DocMaster 实现**：
```
Agent 发现缺少功能 → create_tool 生成代码
→ AST 安全检查 → 沙盒试执行 → 保存 .draft 草稿
→ 用户审批 approve_tool → 重命名为 .py → 注册到 ToolRegistry
→ 下次启动自动加载 → 永久可用
```

**面试亮点**：这是 **Meta-Tool**（元工具）模式——工具可以产生工具，Agent 的能力边界不断扩展。

## 12. LangChain / LangGraph / CrewAI 对比

| 框架 | 特点 | 适用场景 |
|------|------|----------|
| **LangChain** | 通用 AI 应用框架，链式调用 | 快速原型 |
| **LangGraph** | 基于图的状态机，支持循环和分支 | 复杂工作流 |
| **CrewAI** | 角色扮演式多 Agent 协作 | 团队协作模拟 |
| **DocMaster** | 手写轻量框架，领域专精 | 特定领域 Agent |

**手写的优势**：
1. 依赖少——不需要几十个包
2. 完全可控——可以精确控制每一步的行为
3. 面试友好——可以解释每一行代码的设计决策

## 13. Skill 插件化架构

**问题**：System Prompt 不能无限长，所有业务知识都塞进去会超 Token 且互相干扰。

**解法**：按需加载——只在用户输入匹配时才注入对应的 Skill 内容。

**DocMaster 的双层匹配**：
1. Layer 1: 关键词匹配（零 API 调用）
2. Layer 2: Embedding 语义匹配（Layer 1 无命中时触发）

**热插拔**：新增 Skill 只需在 `skills/` 目录加一个 `.md` 文件，零代码变更。

## 14. 语义切块 vs 机械切块

| 维度 | 机械切块 | 语义切块 |
|------|----------|----------|
| 策略 | 固定 500 字一刀 | Embedding 相似度断崖检测 |
| 效果 | 可能拆散同一段落 | 保持语义完整性 |
| 成本 | 零 | 需要 Embedding API 调用 |
| 实现 | `re.split()` | `semantic_chunker.py` |

**语义切块算法**：
1. 按标点切成句子
2. 批量 Embedding
3. 计算相邻句子余弦相似度
4. 自适应阈值 = μ - 1σ
5. 低于阈值处 = 话题边界 = 切块点

## 15. COM 自动化与进程管理

**问题**：Word COM 接口不稳定——调用后进程可能残留，多次调用可能冲突。

**DocMaster 的解法**：
- `com_watchdog.py` 的 `COMSafeLock`：DispatchEx 进程隔离 + PID 精准狙击
- `word_cleanup.py`：Pipeline 结束后强制清理残留进程
- Multi-Agent 的 `_discard_word_changes()`：Pipeline 级别兜底

**面试加分点**：这是真实的工程问题——只有实际操作过 Windows COM 的人才会遇到。
